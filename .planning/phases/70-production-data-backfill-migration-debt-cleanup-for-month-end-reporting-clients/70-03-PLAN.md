<objective>
Backfill `forecast_payroll_summary` for every active forecast across production. Audit found 0 rows on all 3 sampled clients despite Envisage having 6 employees. This means the wages tab roll-up is unwired for the entire production set — even after Phase 71's name-matching fix (B1) lands, there will be no budget side to compare against.

Purpose: `forecast_payroll_summary` is keyed by `forecast_id` (one row per forecast) and stores jsonb monthly maps (`wages_admin_monthly`, `wages_cogs_monthly`, `superannuation_monthly`, `payroll_tax_monthly`, `payg_monthly`, `net_wages_monthly`, `pay_runs_per_month`). The cashflow engine (`src/lib/cashflow/engine.ts`) reads these maps directly. Without them, the wages tab + cashflow's wages line both fall back to zero.

Output: A two-mode script. Default = dry-run preview listing every active forecast that needs a backfill, the computed monthly totals per month, and the deltas. `--apply` = upserts one row per active forecast in `forecast_payroll_summary`.
</objective>

  <action>
Create `scripts/70-03-A2-payroll-summary-backfill.mjs` accepting zero flags (default = dry-run preview, no writes) or `--apply` (commits upserts).

Algorithm:

1. Env load: `config({ path: '.env.local' })` + fallback `config({ path: '.env' })`. Use `SUPABASE_SECRET_KEY` with `SUPABASE_SERVICE_KEY` fallback. Print "DRY RUN" or "APPLY MODE" header.

2. Read all active forecasts (this runs AFTER 70-02 has resolved duplicates, so each business has at most one active forecast per FY): `SELECT id, business_id, name, fiscal_year, forecast_start_month, forecast_end_month FROM financial_forecasts WHERE is_active = true`.

3. For each active forecast `F`:
   a. Read all employees: `SELECT * FROM forecast_employees WHERE forecast_id = F.id AND (is_active IS NULL OR is_active = true)`.
   b. Build the month key array from `F.forecast_start_month` to `F.forecast_end_month` inclusive — both are 'YYYY-MM' strings. Iterate month-by-month (parse year + month, increment, format back to 'YYYY-MM') to produce the array (e.g. `['2025-07', '2025-08', ..., '2026-06']`).
   c. For each month key `M`:
      - Initialize per-month accumulators: `wages_admin = 0, wages_cogs = 0, super = 0, payroll_tax = 0, payg = 0, pay_runs = 0`.
      - For each employee `E`:
        - Active in month: `E.start_date <= M` AND (`E.end_date IS NULL` OR `E.end_date >= M`). String comparison works because both are 'YYYY-MM'.
        - If not active in month: skip.
        - Compute monthly wages: `wages = E.monthly_cost ?? (E.annual_salary != null ? E.annual_salary / 12 : 0)`.
          - If both are null/missing, log a WARN ("employee X in forecast Y has no monthly_cost and no annual_salary — skipping") and skip the employee.
        - Compute super: `superVal = wages * (E.super_rate ?? 0.12)`. (Locked default per CONTEXT A2 2026-05-31: 0.12 / 12% — matches current AU SG statutory rate FY27+ and the wizard's default in `src/app/finances/forecast/components/forecast-cfo/hooks/useForecastCFO.ts`.)
        - Compute PAYG: the locked decision is `paye_amount_monthly ?? wages × 0.32`. ForecastEmployee has `payg_per_period` (pay-period sized) not `paye_amount_monthly`. Convert to monthly: if `E.payg_per_period` is set AND the forecast has a known pay frequency, prorate; otherwise fall back to `wages × 0.32`. SIMPLIFIED FOR BACKFILL: `payg = wages × 0.32` (the explicit fallback is the conservative choice; deferring per-employee provenance to next phase).
        - Compute net wages: `net = wages - payg`.
        - Compute payroll tax: `ptax = wages * 0.0485` (NSW rate per locked decision — flag multi-state as out of scope; document).
        - Compute pay_runs contribution: 4.33 (weekly), 2.17 (fortnightly), or 1.0 (monthly). DEFAULT = 1.0 (monthly) because the per-forecast pay frequency is not stored on ForecastEmployee directly; the existing wizard's pay frequency is a per-forecast config which we cannot reliably read here. DOCUMENT: pay_runs_per_month is set to 1.0 per employee active in month (so total = count of active employees treated as monthly-paid); follow-up phase should distinguish.
        - Add to the right bucket: `if E.classification === 'cogs' then wages_cogs += wages else wages_admin += wages`. Always add super/payg/ptax to their global accumulators (not split by opex/cogs — the schema doesn't split them).
      - Record in maps:
        - `pay_runs_per_month[M] = pay_runs`
        - `wages_admin_monthly[M] = wages_admin`
        - `wages_cogs_monthly[M] = wages_cogs`
        - `superannuation_monthly[M] = super`
        - `payroll_tax_monthly[M] = payroll_tax`
        - `payg_monthly[M] = payg`
        - `net_wages_monthly[M] = wages_admin + wages_cogs - payg`
   d. Read existing `forecast_payroll_summary` row for this forecast_id (may be null per audit).
   e. IDEMPOTENCY: deep-compare the computed maps to the existing row (if any). If identical, mark as "skip (already correct)" — do not increment the "will write" counter.

4. Dry-run output:
```
═══════════════════════════════════════════════════════
forecast_id=<uuid>  business_profile_id=<bp.id>  name="FY27 v1"  range=2026-07..2027-06
  employees: 6 total (4 opex, 2 cogs, 0 inactive)
  Month 2026-07: wages_admin=20833 wages_cogs=8333 super=3500 ptax=1414 payg=9333 net=19833
  Month 2026-08: ... (full 12 lines per forecast)
  ...
  existing row: NONE → will INSERT
  OR
  existing row: present → will UPDATE (delta: wages_admin month 2026-07 changed 0 → 20833)
═══════════════════════════════════════════════════════
```
Print delta detail ONLY for months that differ — full diff is too noisy.

5. If `--apply`:
   - For each forecast: build the full PayrollSummary object, upsert via:
     ```js
     await sb.from('forecast_payroll_summary').upsert({
       forecast_id: F.id,
       pay_runs_per_month, wages_admin_monthly, wages_cogs_monthly,
       payg_monthly, net_wages_monthly, superannuation_monthly, payroll_tax_monthly,
     }, { onConflict: 'forecast_id' })
     ```
   - Per-row try/catch. Log success/failure per forecast_id.

6. Summary at end:
   - "Active forecasts examined: N"
   - "Forecasts needing backfill: M" (where existing row was missing OR maps differed)
   - "Forecasts skipped (already correct): K" (idempotency check passed)
   - "Forecasts upserted: W" (--apply mode only; W must equal M)

Tie-handling / edge cases (document inline):
- Forecast with 0 employees: skip entirely — do not write an empty row (the wages tab handles missing-row case correctly; a row of all zeros is misleading).
- Forecast missing forecast_start_month OR forecast_end_month: log WARN, skip. (Audit shows all production forecasts have these set; defensive only.)
- Employee with annual_salary present but is 0: treat as 0 wages (do not skip — operator intent unclear).
- Employee with end_date earlier than start_date: log WARN, skip.

//   super       = wages × super_rate (default 0.12 / 12% — AU SG statutory rate FY27+)

Avoid (and WHY):
- Do NOT call any code in `src/app/finances/forecast/components/wizard-v4/utils/` — those are React-side TS modules that cannot be imported into a node mjs script without a transpile step. The compute logic is small enough to be inline here. The trade-off (DRY violation now, backfilled values for everyone) is what CONTEXT decisions A2 explicitly accept.
- Do NOT wire this into the forecast save flow — that's the deferred follow-up note in CONTEXT.md. THIS phase is a one-shot backfill, full stop.
- Do NOT split super by opex/cogs — the schema flat-fields super/ptax/payg across both classifications. Wages are the only split.
- Do NOT use `.insert()` then catch unique-violation. Use `.upsert({ ... }, { onConflict: 'forecast_id' })` — idempotent and explicit.
- Do NOT compute for inactive forecasts (per Phase 67 + 70-02 work, only one active per FY exists; building data for inactives wastes I/O and risks confusing downstream).
- Do NOT use 0.115 as the super default — the locked CONTEXT value (per Matt 2026-05-31) is 0.12. If 0.115 appears anywhere in this script, that is a planning/execution bug.
