<objective>
Clean up Envisage's `subscription_budgets` table: (1) dedupe the duplicate Paypal entries by keeping the more specific row and deleting the generic one, transferring any account_codes from the deleted row to the kept row first; (2) backfill empty account_codes on all remaining rows by inferring from Xero P&L vendor activity in `xero_pl_lines_v2`.

Purpose: Empty account_codes mean the monthly report's subscription tab cannot match this vendor's spend to any P&L line — variance is missing from every Envisage subscription. This is the longest-tail data debt for the only sampled client whose other dimensions are mostly healthy.

Output: A two-mode script. Default = dry-run preview showing (a) the Paypal merge plan, (b) per-vendor proposed account_codes (from xero_pl_lines_v2 contact matching). `--apply` = commits the merge + the account_codes updates.
</objective>

Algorithm:

STEP A — Paypal merge:
1. Read all subscription_budgets rows for Envisage WHERE vendor_name ILIKE '%paypal%'. Expect ≥ 2 rows.
2. Identify the "specific" row: vendor_name matches `/Paypal Australia\b/i` OR contains a digit sequence (the bank reference). Identify the "generic" row: vendor_name = 'Paypal' (case-insensitive exact) or vendor_name = 'PayPal'.
3. If exactly 1 specific + 1 generic: merge plan:
   - keeper = specific row
   - loser = generic row
   - merged account_codes = uniq(concat(keeper.account_codes ?? [], loser.account_codes ?? []))
   - merged annual_budget = max(keeper.annual_budget, loser.annual_budget) (the locked decision in CONTEXT says "keep the more specific entry, delete the generic one" — but if generic has a budget and specific doesn't, transfer the budget. Document this in the dry-run output.)
   - merged is_active = keeper.is_active OR loser.is_active
4. If 0 generic / 0 specific / 2+ of either: do NOT mutate; log "Paypal pattern not as expected — manual review" and emit error.
5. Dry-run output:
```
═══ PAYPAL MERGE PLAN ═══
KEEP   id=<uuid>  vendor="Paypal Australia 1043714034893"  budget=$X  codes=[...]
DELETE id=<uuid>  vendor="Paypal"                          budget=$Y  codes=[...]
MERGED row will have: budget=$max(X,Y) codes=uniq(...) is_active=true
```
6. On --apply: (in sequence, never bulk)
   a. UPDATE keeper to merged values
   b. DELETE loser  ← this IS a delete; the CONTEXT explicitly says "delete the generic one"
   c. Per-step try/catch; if (b) fails after (a), log loudly and do NOT proceed to step B.

STEP B — account_codes backfill:
1. Read all Envisage subscription_budgets rows where `is_active=true` AND `account_codes IS NULL OR account_codes = '{}'`.
2. For each row, attempt to infer account_codes from Xero P&L data:
   a. Build the vendor key: `target_key = createVendorKey(row.vendor_name)` (import from src/lib/utils/vendor-normalization.ts — same as 70-04; this script is .ts).
   b. Query `xero_pl_lines_v2` for the LAST 12 MONTHS for Envisage (`business_id = ENVISAGE_PROFILES_ID`). Select per-line: `account_code, account_name, contact_name` (or whichever column carries vendor identity — verify against the latest migration; if `contact_name` doesn't exist on `xero_pl_lines_v2`, check `xero_pl_lines` or fall back to live Xero BankTransactions per-tenant for Envisage).
   c. For each PL line: `tx_key = createVendorKey(line.contact_name)`. If `tx_key === target_key`, record the `account_code`.
   d. Inferred account_codes = the most frequent N codes (capped at 3 — most subs hit 1-2 accounts; >3 is noise).
3. If `inferred.length > 0`: propose UPDATE `account_codes = inferred` (sorted, deduped).
4. If `inferred.length === 0`: mark as "unresolved (no PL activity matched)". Surface for Matt's review — but do NOT auto-fill.
5. Dry-run output:
```
═══ ACCOUNT_CODES BACKFILL ═══
  ✓ "Mailchimp"     → codes=['612']   (3 matches in last 12mo, all account 612)
  ✓ "Adobe"          → codes=['612','620']  (mixed; 612 dominant)
  ⚠ "LinkedIn"       → no PL matches (zero activity)
  ...
```

6. On --apply: per-row UPDATE subscription_budgets SET account_codes = X WHERE id = Y. Per-row try/catch.
