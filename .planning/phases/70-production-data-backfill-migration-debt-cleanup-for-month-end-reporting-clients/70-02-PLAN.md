Create `scripts/70-02-A1-active-forecast-remediation.mjs` accepting zero flags (default = dry-run preview, no writes) or `--apply` (commits writes).

Algorithm:

1. Env load: `config({ path: '.env.local' })`, then fallback `config({ path: '.env' })`. Use `SUPABASE_SECRET_KEY` with `SUPABASE_SERVICE_KEY` fallback. Print "DRY RUN — preview only" or "APPLY MODE — writes will commit" prominently at top per `onboard-fit2shine.mjs` line 331-332 pattern.

2. Read ALL `financial_forecasts` rows with `is_active=true` across ALL businesses (NO business filter). Select: `id, business_id, name, fiscal_year, year_type, version_number, updated_at, created_at`.

3. Group by `(business_id, fiscal_year, year_type)` composite key. (year_type included because FY and CY at the same fiscal_year number could legitimately coexist; the Phase 67 constraint is on the same year_type.)

4. For each group with `size > 1`:
   a. For each forecast in the group, additionally fetch:
      - `pl_line_count`: `SELECT count(*) FROM forecast_pl_lines WHERE forecast_id = X` via `.select('id', { count: 'exact', head: true })`
      - `payroll_summary_nonzero`: `SELECT 1 FROM forecast_payroll_summary WHERE forecast_id = X AND wages_admin_monthly::text != '{}' AND wages_admin_monthly IS NOT NULL` — but since the column is jsonb, simpler: fetch the row and check if any key has a positive value. If the row exists at all, treat as "has payroll" = 1, else 0. (Audit confirms most rows are missing entirely.)
   b. Apply the locked sort (winner = first):
      - ORDER BY updated_at DESC, pl_line_count DESC, payroll_summary_nonzero DESC, created_at DESC
   c. Winner = first; losers = the rest.

5. Dry-run output (always print, even in --apply mode for the audit log):
```
═══════════════════════════════════════════════════════
business_id=<bp.id>  fiscal_year=2026  year_type=FY
  ✓ WINNER  id=<uuid>  name="..."  updated=<iso>  pl_lines=92  payroll=1  created=<iso>  reason="latest updated_at"
  ✗ LOSER   id=<uuid>  name="..."  updated=<iso>  pl_lines=0   payroll=0  created=<iso>  → will set is_active=false
  ✗ LOSER   id=<uuid>  ...
═══════════════════════════════════════════════════════
```
For each loser, include the explicit `reason` field on the winner line (e.g., "latest updated_at" vs "tie on updated_at; more pl_lines" vs "tie on pl_lines; has payroll" vs "tie on payroll; latest created_at").

6. If `--apply`:
   - For each loser: `UPDATE financial_forecasts SET is_active = false, updated_at = NOW() WHERE id = <loser.id>`. Do these ONE AT A TIME (not bulk) so a single bad ID does not nuke the batch. Wrap each in try/catch; print `  ✓ deactivated <id>` or `  ✗ FAILED <id>: <err>`.
   - Track total mutations attempted vs succeeded.
   - Do NOT delete any row. Do NOT touch the winner. Do NOT touch any other column.

7. Summary at end:
   - "Groups with conflicts: N"
   - "Losers identified: M"
   - "Losers deactivated: K" (only meaningful in --apply mode)
   - If `K !== M` in --apply mode: print loud error and `process.exit(1)`.

8. Idempotency: after --apply, re-running with no flags must show "Groups with conflicts: 0". Verify this manually in Task 2.

Tie-breaker edge cases (document inline as comments):
- If TWO losers have identical updated_at + identical pl_line_count + identical payroll status + identical created_at: this shouldn't happen in production, but if it does, deactivate ALL but one (pick the one with the lowest id alphabetically as a deterministic final fallback). Log this case as a WARN.
- If a group has all rows with `pl_line_count=0` AND `payroll_summary_nonzero=0` (truly empty): still pick the most-recently-updated as winner — do not deactivate everything (Phase 67 allows zero active, but having zero would break the wizard load path; the audit explicitly calls this out at CONTEXT decisions A1 paragraph 4).

Avoid (and WHY):
- Do NOT delete forecast rows. The audit history matters; "deactivate not delete" is locked in CONTEXT.md.
- Do NOT use a single bulk `UPDATE ... WHERE id IN (...)` — per-row update with per-row error capture is the safer pattern. Speed is not the constraint.
- Do NOT skip the secondary tie-breakers in dry-run output. The point of dry-run is to show Matt the rationale before he approves.
- Do NOT touch forecasts for businesses outside production (e.g., archived demo orgs). If a business has `status != 'active'` in the `businesses` table, log it but do NOT mutate. Resolve by adding a join on `businesses.status='active'` upstream — but since financial_forecasts.business_id keys business_profiles.id and not businesses.id, the join goes via `business_profiles.business_id → businesses.id`. Print "skipping (parent business not active)" so the dry-run output is transparent.
