    ### Criterion A: Legitimate denies only
    Every `verdict_reason: 'permission_denied'` event must come from a user who IS a `business_users` member with:

    WHERE business_id = '<from event extra>' AND user_id = '<from event extra>';
    ```

    ### Criterion B: No unexpected `not_a_member` for known-good users
    `verdict_reason: 'not_a_member'` is expected for legitimately unauthenticated or non-member call attempts (e.g. random bot traffic, mistyped business IDs). It is NOT acceptable if it comes from:
    - A business owner (cross-check `businesses.owner_id`)
    - An assigned coach
    - A super_admin (cross-check `system_roles`)

    ```sql
    SELECT 'owner' as via FROM businesses WHERE id='<bid>' AND owner_id='<uid>'
    UNION ALL
    SELECT 'system_role' as via FROM system_roles WHERE user_id='<uid>' AND role='super_admin'
    -- plus whatever the existing coach-assignment query is (mirror 65-01 helper code)
    ;
    ```
    Any row returned for an event tagged `not_a_member` → the helper missed an allow case. Soak FAILS.

    - Total `section_permission_check` events ≥ 1 (otherwise we have no signal — extend soak or investigate why no member-status calls are happening). If the production tenant has no `finances:false` members at all, this criterion auto-passes — note it in the readout.
    - No order-of-magnitude burst (e.g. 100,000 events in 1h) — indicates a broken loop or a feature that's hammering the endpoint. Investigate before flipping.
