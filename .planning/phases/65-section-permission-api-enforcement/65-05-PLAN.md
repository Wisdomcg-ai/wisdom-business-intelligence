    ```markdown
    # Phase 65 — PR risk assessment + rollback recipe

    **Phase:** 64 section-permission API enforcement
    **Date completed:** {YYYY-MM-DD}
    **Production state at sign-off:** `SECTION_PERMISSION_ENFORCE = true` in Vercel production

    ---

    ## What changed for end users

    A `business_users` member with `section_permissions.finances = false` who attempts to call a finance-gated API route directly (e.g. via browser devtools, curl, or a script) now receives:

    ```
    HTTP/1.1 403 Forbidden
    Content-Type: application/json
    { "error": "Insufficient permissions", "section": "finances" }
    ```

    Previously they received a 200 with full data. The UI (sidebar filter) already hid these routes from them, so a normal session of clicking around the app sees no change. The change blocks the bypass-via-devtools path.

    Owners, admins, assigned coaches, super_admins, and members with `finances: true` see no change.

    ## Failure modes considered + mitigations

    | Risk | Likelihood | Mitigation |
    |------|-----------|------------|
    | Helper denies a legitimate user (missing allow case) | Low (mitigated by 65-03 soak; helper allowlist matches `auth_can_access_business` semantics) | Kill switch (below) reverts to LOG_ONLY in <5 min |
    | Section-key spelling mismatch (`financials` vs `finances`) | Medium → Resolved | Wave 65-01 locked `finances` as canonical with grep evidence in `65-01-SECTION-KEY-VERIFICATION.md`; spelling guard test pins the decision |
    | A finance route was missed from the inventory | Low | Wave 65-02 used `find src/app/api -name route.ts` enumeration; 65-03 soak detects orphan denies/allows; new finance routes added in future phases must include the helper call (add to project skill or convention doc) |
    | Service-role bypass | Not applicable | Grep-asserted zero `createServiceRoleClient` references in all edited routes |
    | Helper exception on Supabase failure | Medium | Helper throws; existing route try/catch wrappers surface 5xx. LOG_ONLY soak in 65-03 caught zero such events |
    | Env var doesn't propagate to runtime | Low | Read at module load on every Lambda cold start; `vercel --prod` redeploy is required after `vercel env add` (documented in 65-04 runbook) |
    | UI shows stale data after a member is locked out | Not applicable | UI already gated; user never saw the data |
    | Other section keys (`business_plan`, `execute_*`) | Out of scope | Phase 65 finance only. Future phase extends to other keys using same helper |

    ## Rollback recipe (kill switch — <5 minutes)

    If a legitimate user reports a 403, or any unexpected behavior surfaces:

    ```bash
    # 1. Remove the env var (defaults to false → LOG_ONLY)
    vercel env rm SECTION_PERMISSION_ENFORCE production
    # Confirm with 'y'

    # 2. Redeploy to pick up the change
    vercel --prod

    # 3. Confirm in Sentry: filter for message:"section_permission_check" environment:production
    #    Within 5 min of the new deploy going live, every new event should have enforced:false (was true before).
    ```

    Success criterion: zero 403 responses on finance routes within 5 min of redeploy AND new Sentry events show `enforced: false`.

    If env-var rollback is insufficient (extremely unlikely):

    ```bash
    git revert <65-02-merge-commit-sha>
    git push origin main
    # CI + auto-deploy, or `vercel --prod` to force
    ```

    This removes the helper calls from every route, returning to pre-phase-64 behavior.

    ## Test coverage shipped

    - Unit tests: `src/lib/permissions/__tests__/requireSectionPermission.test.ts` — 11 cases covering every allow/deny verdict + the `finances` spelling guard.
    - LOG_ONLY integration tests: 3 routes (monthly-report/generate, forecast/[id], Xero/pl-summary).
    - ENFORCE-mode integration tests: same 3 routes, with regression-guard test pinning the LOG_ONLY rollback path.

    Total new test files: 7. All pass under `npx vitest run`.

    ## What's NOT in this phase (deferred)

    - Other section keys (`business_plan`, `execute_*`, `coaching_*`). Future phase.
    - Coach `/api/coach/**` routes (already allow-listed by `auth_can_access_business`; covered if we extend the helper to them later).
    - Cleanup of the baseline migration's legacy `financials` JSONB default — would need a migration. Phase 66+.
    - Audit ops/admin routes that use service-role bypass.

    ## Links

    - Phase context: [`65-CONTEXT.md`](./65-CONTEXT.md)

    - Cutover runbook: [`65-04-ENFORCE-CUTOVER.md`](./65-04-ENFORCE-CUTOVER.md)
    - Phase summary: [`65-SUMMARY.md`](./65-SUMMARY.md)
