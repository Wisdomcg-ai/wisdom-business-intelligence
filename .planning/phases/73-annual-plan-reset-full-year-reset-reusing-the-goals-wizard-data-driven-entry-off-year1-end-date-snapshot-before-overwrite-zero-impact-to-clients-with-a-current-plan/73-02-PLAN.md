<objective>
Implement the rollover action that turns a finished plan year into a fresh one IN the goals data
model: shift the 3-year ladder forward (prepopulate new Year1 from prior Year2 per D3), roll the
plan dates to the new FY/CY, clear quarterly_targets so the wizard's own Step4 even-split defaults
apply, and carry incomplete initiatives forward as candidates. The whole operation is gated on a
73-01 snapshot succeeding first (snapshot-before-overwrite).

Purpose: This is the mutation core. It must be deterministic (pure math, unit-tested at the FY and
CY boundary) and safe (no write without a prior snapshot).

Output: pure `rollover-math.ts` + an `AnnualResetService.executeAnnualReset(...)` orchestrator,
both fully tested.
</objective>

LADDER COLUMNS to shift in business_financial_goals (keyed on business_profiles.id). For EACH of
these metric prefixes, columns are <prefix>_current/_year1/_year2/_year3:
  revenue, gross_profit, gross_margin, net_profit, net_margin, customers, employees,
  leads_per_month, conversion_rate, avg_transaction_value, team_headcount, owner_hours_per_week
ROLLOVER RULE (per D3):
  new_current = prior_year1   (the year just finished becomes the baseline)
  new_year1   = prior_year2
  new_year2   = prior_year3
  new_year3   = prior_year3   (extrapolate: carry Year3 forward; client adjusts in wizard)

PLAN DATE roll (honour Phase-42 explicit dates + Phase-14 extended semantics). Reuse
fiscal-year-utils + suggest-plan-period:
```typescript
// src/app/goals/utils/suggest-plan-period.ts (existing, pure):
suggestPlanPeriod(today: Date, yearStartMonth = 7): { planStartDate, planEndDate, year1EndDate, year1Months, rationale }
// src/lib/utils/fiscal-year-utils.ts:
getFiscalYearStartDate(fy, yearStartMonth), getFiscalYearEndDate(fy, yearStartMonth), getFiscalYear(date, yearStartMonth)
```
NEW PLAN DATES: the new Year1 = the FY that begins immediately after the prior year1_end_date.
  newPlanStart = day after prior year1_end_date (== getFiscalYearStartDate(newFY, ysm))
  newYear1End  = getFiscalYearEndDate(newFY, ysm)
  newPlanEnd   = getFiscalYearEndDate(newFY + 2, ysm)   // +3 years total (Y1,Y2,Y3)
  is_extended_period = false, year1_months = 12, current_year_remaining_months = 0  (reset is a clean standard year)
year_type is PRESERVED from the prior row ('FY' or 'CY').

quarterly_targets RESET: set to {} (empty jsonb). The wizard's Step4AnnualPlan applies its existing
even-split-from-annual defaults when quarterly_targets is empty — DO NOT invent new defaults here (locked).

INITIATIVES carry-forward (strategic_initiatives, keyed on businesses.id):
  status CHECK allowed: 'not_started'|'in_progress'|'completed'|'cancelled'|'on_hold'
  INCOMPLETE = status IN ('not_started','in_progress','on_hold').
  Carry forward = UPDATE those rows: status='not_started', selected=false (candidate, client opts in),
  fiscal_year=newFY, updated_at=now. Leave step_type as-is. DO NOT touch completed/cancelled rows.
  (Mirrors the existing carry-forward UPDATE in strategic-sync-service.ts syncAnnualReview Part B,
   but selected=false so it's a candidate, not auto-committed — per locked decision.)

EXISTING SUPABASE WRITE PATTERN: createClient() from '@/lib/supabase/client', .update(payload).eq('business_id', ...).
There is NO multi-statement DB transaction in the browser client; "transactional boundary" here means:
ordered, snapshot-gated, fail-fast — abort remaining writes and surface error if any step fails after snapshot.
