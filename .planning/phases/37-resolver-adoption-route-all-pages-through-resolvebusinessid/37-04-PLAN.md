<objective>
Route the strategic-planning surface (one-page-plan + goals wizard +
operational plan tab + business-dashboard hook + plan data assembler)
through `resolveBusinessId`. This is the trickiest cluster because several
files need the resolved `businesses.id` translated to `business_profiles.id`
for downstream queries — the same two-step pattern Plan 01 Task 3 handled
for useDashboardData.

Purpose: Eliminate duplicate resolution logic in the strategic-planning
data path while preserving the three-ID architecture documented in
useStrategicPlanning (user.id for SWOT, businesses.id for coach-client
links, business_profiles.id for goals/initiatives/KPIs).

Output: Six files using `resolveBusinessId` with zero inline
`eq('owner_id', user.id)` lookups.
</objective>

    NEW `else if` branch:

    ```ts
    } else if (userRole === 'client' || userRole === undefined) {
      // Client view — resolve via helper (handles team-membership + owner chains).
      const { resolveBusinessId } = await import('@/lib/business/resolveBusinessId')
      const resolved = await resolveBusinessId(supabase, {
        userId: user.id,
        role: 'client',
        activeBusinessId: null,
      })

      if (!resolved.businessId) {
        // No business → client onboarding state, bail.
        return null
      }

      businessesId = resolved.businessId

      // Look up owner_id for SWOT (SWOT is keyed on user.id — the OWNER's user.id,
      // not the current user's — so for team members we need the owner).
      const { data: bizData } = await supabase
        .from('businesses')
        .select('owner_id')
        .eq('id', businessesId)
        .maybeSingle()
      ownerUserId = bizData?.owner_id || user.id

      // Translate businesses.id → business_profiles.id.
      const { data: profileData } = await supabase
        .from('business_profiles')
        .select('id, industry, owner_info, key_roles')
        .eq('business_id', businessesId)
        .single()

      profile = profileData
      if (!profile?.id) return null
      businessId = profile.id
    } else {
      // coach/admin with no active client — bail
      return null
    }
    ```

    Prefer a static import at top of file over dynamic import if it doesn't cause circular-dep issues:

    ```ts
    import { resolveBusinessId } from '@/lib/business/resolveBusinessId'
    ```

    The `activeBusiness?.id` branch at the top of the if/else ladder stays UNCHANGED — it already resolves correctly and the resolver would return the same value (reason: 'active').

    **Behaviour note:** The old code looked up profile by `user_id` (which worked for owners only). The new code looks up profile by `business_id` after resolving the `businesses.id` — this correctly handles team members who have a business_user row but NOT a business_profile.user_id match. This is an IMPROVEMENT in correctness, not a regression.

    Leave every subsequent data-loading block (goals, initiatives, KPIs, SWOT, quarters) completely unchanged.
