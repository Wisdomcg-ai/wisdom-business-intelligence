<!-- Each existing endpoint's current shape -->

`/api/Xero/pl-summary`: returns wide-shaped account rows with monthly_values + variance vs forecast. Pre-44 logic: SELECT from xero_pl_lines, JOIN forecast_pl_lines, compute variance in JS. Post-44: delegate to getMonthlyComposite + compute variance from the composite.

`/api/forecast/cashflow/xero-actuals`: returns cash-projection-shaped data. Pre-44 logic: SELECT, classify, sum. Post-44: delegate to getCashflowProjection.

`/api/monthly-report/generate`: returns current-month variance for the report. Pre-44: SELECT + categorize + variance per row + subtotals. Post-44: delegate to getCategorySubtotalsForMonth + getMonthlyComposite.

`/api/monthly-report/full-year`: 12-month YTD/forecast comparison. Same pattern.

<!-- Boundary rule -->

Each migrated endpoint MUST:
1. Authenticate the user.
2. Resolve forecastId (from query param OR look up the active forecast for the requested business_id + fiscal_year via the unique_active_forecast_per_fy contract).
3. Call ForecastReadService method.
4. If the service throws an invariant violation, surface as a 500 with structured error body — do NOT silently fall back to old logic.
5. Return the SAME JSON shape the legacy endpoint returned (consumer UI components must not need changes).
