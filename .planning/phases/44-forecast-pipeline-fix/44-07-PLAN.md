<!-- Recompute endpoint contract -->

`POST /api/forecast/[id]/recompute` (NEW):
- Auth: authenticated user must have access to forecast (via standard business membership check).
- Body: empty (just an action trigger; no parameters needed).
- Behavior:
  1. Load forecast_assumptions.data for the forecast.
  2. Call computeForecastLinesFromAssumptions(...) → pl_lines array (same JS function the wizard uses).
  3. Call supabase.rpc('save_assumptions_and_materialize', ...) with the SAME assumptions (idempotent re-write) + freshly derived pl_lines.
  4. Return { forecast_id, computed_at, lines_count }.

This is the recovery hatch for: (a) forecasts created before Phase 44 that have stale or missing computed_at, (b) derivation logic changes after this plan ships, (c) one-off remediation when an assumption save somehow shipped non-atomically (shouldn't happen post-RPC, but the hatch exists).

    ```typescript
    import { NextRequest, NextResponse } from 'next/server'
    import { createRouteHandlerClient } from '@/lib/supabase/server-client'  // match the project's actual import
    import { resolveBusinessIds } from '@/lib/utils/resolve-business-ids'
    import { computeForecastLinesFromAssumptions } from '@/lib/forecast/derive-pl-lines'  // ADJUST: use whatever helper the wizard's generate route uses for derivation

    export const dynamic = 'force-dynamic'

    /**
     * Phase 44 D-12 — recovery hatch.
     *
     * Re-derive forecast_pl_lines for a given forecast using the CURRENT assumptions row
     * (no input from caller). Useful for:
     *   1. Forecasts created before Phase 44 that have stale or missing computed_at.
     *   2. After derivation logic changes (deploy a new computeForecastLinesFromAssumptions; trigger this for affected forecasts).
     *   3. Manual remediation when ForecastReadService throws an invariant violation.
     *
     * Auth: user must have access to the forecast's business (via business_users, owner_id, or coach role).
     * Body: none.
     * Returns: { forecast_id, computed_at, lines_count } from the RPC.
     */
    export async function POST(
      req: NextRequest,
      ctx: { params: Promise<{ id: string }> }
    ) {
      const { id: forecastId } = await ctx.params
      const supabase = createRouteHandlerClient()

      // 1. Auth.
      const { data: { user } } = await supabase.auth.getUser()
      if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })

      // 2. Verify forecast access. Load the forecast → its business_id → check user has access via the standard pattern.
      const { data: forecast, error: fError } = await supabase
        .from('financial_forecasts')
        .select('id, business_id')
        .eq('id', forecastId)
        .maybeSingle()
      if (fError || !forecast) return NextResponse.json({ error: 'Forecast not found' }, { status: 404 })

      const ids = await resolveBusinessIds(supabase, forecast.business_id)
      const { data: access } = await supabase
        .from('business_users')
        .select('user_id')
        .eq('user_id', user.id)
        .in('business_id', ids.all)
        .limit(1)
        .maybeSingle()
      const isOwner = await supabase
        .from('businesses')
        .select('id')
        .eq('owner_id', user.id)
        .in('id', ids.all)
        .limit(1)
        .maybeSingle()
        .then(r => !!r.data)
      // Allow coach/admin via existing system_roles pattern — copy the exact check from generate/route.ts
      // (ADJUST: paste the exact role-check helper used elsewhere in the codebase)
      if (!access && !isOwner /* && !isCoachOrAdmin */) {
        return NextResponse.json({ error: 'Forbidden' }, { status: 403 })
      }

      // 3. Load current assumptions.
      const { data: assRow, error: aError } = await supabase
        .from('forecast_assumptions')
        .select('data')
        .eq('forecast_id', forecastId)
        .maybeSingle()
      if (aError) return NextResponse.json({ error: aError.message }, { status: 500 })
      const assumptions = assRow?.data ?? {}

      // 4. Derive pl_lines from current assumptions (same JS helper the wizard uses).
      const derivedLines = await computeForecastLinesFromAssumptions(assumptions, { forecastId, supabase })

      // 5. Atomic write via RPC.
      const { data: rpcResult, error: rpcError } = await supabase.rpc('save_assumptions_and_materialize', {
        p_forecast_id: forecastId,
        p_assumptions: assumptions,    // unchanged — re-write is idempotent
        p_pl_lines: derivedLines,
      })
      if (rpcError) return NextResponse.json({ error: `Recompute failed: ${rpcError.message}` }, { status: 500 })

      return NextResponse.json({ success: true, ...rpcResult })
    }
    ```

Read `src/app/api/forecast-wizard-v4/generate/route.ts` to find the exact derivation function used there, and import it from the same location into the recompute route.
