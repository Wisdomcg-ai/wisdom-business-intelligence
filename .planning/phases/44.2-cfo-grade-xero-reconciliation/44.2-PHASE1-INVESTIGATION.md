## Section 1 — Sync orchestrator structure

### 1.1 High-level flow

**File:** `src/lib/xero/sync-orchestrator.ts`

- Entry: `syncBusinessXeroPL(businessId, opts)` at line 232.
- Single-flight via `supabase.rpc('begin_xero_sync_job', { p_business_id: profileId })` at line 246–249.
- Two FY windows (current FY YTD + prior FY) computed at lines 301–319.
- Per-tenant loop at line 358 (`for (const conn of connections)`), inner per-window loop at line 411 (`for (const window of fyWindows)`).
- Per (tenant, window): one `byMonthUrl()` fetch (line 414), one `fyTotalUrl()` fetch (line 426), `parsePLByMonth` + `parseFYTotalResponse` + `augmentWithResiduals` + `reconcilePL` (lines 452–489), `xero_pl_lines` upsert (line 514–519), coverage compute (line 530).
- 300ms `sleep` between Xero requests (line 423, 438).

**byMonthUrl (line 184):** `fromDate=${base.start}&toDate=${base.end}&periods=${periods}&timeframe=MONTH&standardLayout=false&paymentsOnly=false`

**fyTotalUrl (line 204):** `fromDate=${getFYStart(fy, fyStartMonth)}&toDate=${toDate}&standardLayout=false&paymentsOnly=false` (no `periods=`, no `timeframe=`)

**Classification:** **DEFINITELY WRONG** for Path A purposes. The `byMonthUrl` aggregator produces values that don't match Xero web. Path A replaces the by-month call with N per-month single-period calls.

### 1.2 begin/finalize RPC contract

**File:** `supabase/migrations/20260428000005_sync_jobs_state_guard_rpcs.sql`

- 15-minute single-flight window enforced at the business level.
- `finalize` validates `status IN ('success','partial','error')`.

**Classification:** **SAFE.** No change required.

### 1.3 Per-tenant try/catch isolation (44.2-02)

- lines 391 (`try {`) — 608 (`continue`).
- Marks tenant 'error' with bounded message (line 587).
- Sentry capture (lines 595–602).
- Does NOT re-throw (line 607).

**Classification:** **SAFE.** Path A's per-month loop must live inside this try block.

### 1.4 xero_pl_lines upsert + ON CONFLICT

**Evidence (sync-orchestrator.ts:513–524):**
```ts
.upsert(dbRows, {
  onConflict: 'business_id,tenant_id,account_code,period_month',
  ignoreDuplicates: false,

```

DB constraint: `xero_pl_lines_natural_key_uniq (business_id, tenant_id, account_code, period_month)`.

**Classification:** **AT RISK.** `account_code` is actually the Xero AccountID GUID (see §7); naming is misleading.

### 1.5 sync_jobs status / reconciliation JSONB

- Per-tenant rows inside the loop (lines 366–381) with tenant_id populated.
- Terminal update sets reconciliation JSONB shape `{ tenant_id, status: 'mismatch'|'ok', discrepant_accounts: [...] }`.
- Outer business-level row finalized via RPC (lines 683–700).

**Classification:** **SAFE.** Two-tier audit. Post-Path-A this should produce `{ status: 'ok' }` for all healthy syncs.

### 1.6 Cron schedule

**`vercel.json` line 7–9:** `{ "path": "/api/cron/sync-all-xero", "schedule": "0 16 * * *" }`
- 16:00 UTC = 02:00 AEDT / 03:00 AEST
- `maxDuration = 300s`

**Classification:** **AT RISK at scale.** Per-tenant time goes from ~2s to ~13s. Fine for 10 tenants, tight at 25+.

## Section 3 — xero_pl_lines schema & dependencies

### 3.1 Final schema (after 5 migrations)

```
xero_pl_lines:
  id            uuid PK
  business_id   uuid NOT NULL                    -- always business_profiles.id
  tenant_id     text NOT NULL DEFAULT ''
  account_code  text NULL                        -- actually Xero AccountID GUID
  account_name  text
  account_type  text
  section       text                              -- unused in Phase 44+
  period_month  date NOT NULL
  amount        numeric(18,2) NOT NULL DEFAULT 0
  source        text DEFAULT 'xero'
  notes         text NULL                         -- 44.2-06 absorber audit
  created_at    timestamptz DEFAULT now()
  updated_at    timestamptz DEFAULT now()

UNIQUE: xero_pl_lines_natural_key_uniq (business_id, tenant_id, account_code, period_month)
RLS:    xero_pl_lines_access (super_admin OR business_id IN auth_get_accessible_business_ids())
        xero_pl_lines_service_role (service_role bypass)
NO FKs to businesses/business_profiles.
```

## Section 7 — Xero AccountID (GUID) — current state

### 7.1 Where AccountID is captured today

- Parser extracts AccountID from `Cells[0].Attributes` where `Id='account'` (pl-by-month-parser.ts:248–251).
- Stored in `ParsedPLRow.account_code` (misleading name).
- Reconciler does the same (pl-reconciler.ts:336–340).

### 7.2 xero_accounts catalog table (already exists!)

**Evidence (baseline_schema.sql:5523–5539):**
```sql
CREATE TABLE xero_accounts (
  id uuid PRIMARY KEY,
  business_id uuid NOT NULL,
  xero_account_id text NOT NULL,    -- the GUID, named correctly here
  account_code text,                 -- the user-facing 200/300/400
  account_name text NOT NULL,
  xero_type text, xero_class text, xero_status text, ...
);
UNIQUE (business_id, tenant_id, xero_account_id)
```

**Populated by:** `src/app/api/Xero/chart-of-accounts-full/route.ts:79–80` via `/api.xro/2.0/Accounts`.

**Classification:** **SAFE & strategic asset.** Calxa-style chart-of-accounts catalog already exists. Just not wired into the sync orchestrator.

### 7.3 Smallest path

1. **No new fetch needed:** AccountID flows through P&L responses already.
2. Optional enhancement: pull `/Accounts` once per sync to refresh `xero_accounts` for `account_id → account_code → xero_type` mapping.
3. Parser change: rename `ParsedPLRow.account_code` → `account_id`.
4. DB change: add `account_id uuid` column to `xero_pl_lines`; backfill from existing `account_code`; migrate natural key.
