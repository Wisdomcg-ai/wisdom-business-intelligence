<domain>
## Phase Boundary

Establish a CFO-grade data integrity contract: the numbers our system displays MUST equal the numbers Xero displays, exactly, every account, every month, every tenant. This is the platform's foundational invariant — coaches use these numbers in client conversations; "close enough" is reputational risk.

This phase exists because the cross-check against JDS's Xero P&L (2026-04-28 evening) revealed two distinct integrity failures:

**Failure 1 — Account classification.** "Software Development - PK Costs" is **Cost of Sales** in Xero ($81,885 in FY2025) but **Operating Expenses** in our DB ($75,385). Single account, wrong bucket, also wrong amount. Likely cause: parser's parent-section inheritance when Xero returns nested custom-layout sections, OR `standardLayout=false` API param returns a different tree shape than the user-facing report.

**Failure 2 — Amount totals don't reconcile.** Even for accounts in the right bucket, totals differ. JDS FY2025: Total COGS = $6,097,303 in Xero vs $5,737,524 in our DB (off by $359,779). Pattern: by-month sum != FY single-period total for the same account, same date range. Specific example: "Sales - Hardware" FY total $1,897,534.45 but monthly_sum $1,890,695.05, diff $6,839.40, stable across syncs all day.

**Critical pre-existing infrastructure:** Phase 44 Sub-A already built `src/lib/xero/pl-reconciler.ts` (D-08 fail-loud reconciler). It IS running on every sync. It IS catching the $6,839 discrepancy. It writes `sync_jobs.status = 'partial'` when reconciliation fails. **What's missing is the gate at the read path** — the wizard reads `xero_pl_lines` regardless of `sync_jobs.status`, so partial-sync data flows through to the UI as if validated.

After 44.2 ships:
- The wizard, monthly report, cashflow forecast, and dashboard refuse to show data from any sync that didn't reconcile cleanly.
- Coaches see "Data verification in progress" instead of wrong numbers.
- Per-tenant reconciliation is first-class (consolidated entities work correctly).
- The classification + amount gaps in JDS data are diagnosed and either fixed (if our bug) or compensated for (if Xero quirk).
- A continuous reconciliation cron (separate from sync) verifies data integrity against Xero's API daily and alerts on drift.

This is the foundation Phase 44 Sub-A intended but didn't fully wire through.

</domain>

<decisions>
## Implementation Decisions (locked in-session with user)

### Foundational principle (locked 2026-04-28 evening)
- **D-44.2-00:** Xero's ProfitAndLoss API IS the source-of-truth contract. Our system mirrors Xero exactly. Any divergence is a bug, no exceptions, no rounding tolerance beyond $0.01 (which is the existing reconciler tolerance from Phase 44 Sub-A). Coaches using our platform must be able to point at any number and have it match the same number in Xero — every account, every month, every tenant.

### Reconciliation gate (Failure 2 root)
- **D-44.2-01:** `sync_jobs.status = 'success'` is the precondition for any user-facing data display. `partial` and `failed` block reads.
- **D-44.2-02:** When the latest `(business_id, tenant_id)` sync status is `partial` or `failed`, the wizard / monthly report / cashflow / dashboard show a banner: "Data verification in progress — last sync at {timestamp} failed reconciliation against Xero. Click to view detail or trigger re-sync." Numbers are NOT shown, OR shown faded with a warning overlay. Decision: faded with overlay (less alarming than blocking entirely) — but the discrepancy detail is one click away. Plan-checker confirms via UX review.
- **D-44.2-03:** Read service (`ForecastReadService`) returns a `data_quality: 'verified' | 'partial' | 'failed' | 'no_sync'` field on every read. UI consumers honor this field; tests assert presence on every public method.
- **D-44.2-04:** Per-tenant reconciliation is first-class. For consolidated entities (one business, multiple Xero connections), each tenant has its own reconciliation status. The business-level data_quality is the WORST of all tenant statuses.

### Multi-tenant audit trail (Failure 2 derivative)
- **D-44.2-05:** `sync_jobs.tenant_id` MUST be recorded on every sync_jobs write. Currently NULL — that's a multi-tenant audit gap. Backfill plan: leave existing NULL rows alone (they're history); enforce non-null on new writes via NOT NULL constraint after backfill.
- **D-44.2-06:** Reconciliation discrepancy details (the JSONB `reconciliation` column on `sync_jobs`) are the audit trail. They MUST include: `tenant_id`, `account_code`, `account_name`, `fy_total`, `monthly_sum`, `diff`, `period_window`. Already partially in place per Phase 44 Sub-A; verify and extend.

### The $6,839.40 gap (Failure 2 specific diagnosis)
- **D-44.2-07:** Capture both raw Xero responses for JDS (FY single-period query + 12 monthly queries) and store as fixtures in `src/__tests__/xero/fixtures/jds-recon-2026-04.json`. Diff them by hand to locate the missing $6,839.40 on "Sales - Hardware". Most likely candidates (in probability order): year-end adjustment journals dated outside any month bucket; tracking-category aggregation differences; manual journals with non-standard dates; FX revaluation entries.
- **D-44.2-08:** If the gap is a Xero API quirk (by-month and FY queries genuinely return different aggregations for the same data), our reconciler MUST reflect Xero's FY total as the canonical truth — that's what coaches see in Xero's web report. The by-month parser must be made consistent with the FY total either by: (a) adjusting the by-month bucket boundaries, OR (b) injecting an "adjustment" synthetic row that absorbs the gap with a clear audit note.
- **D-44.2-09:** If the gap is our parser bug (by-month parser dropping a row Xero returned), fix the parser. Tests assert exact equality.
- **D-44.2-10:** Decision rule: capture the data first (D-44.2-07), then decide between (a) and (b) based on what the diff shows. No premature decision.

### The classification bug (Failure 1)
- **D-44.2-11:** `classifyAccountType` (in `pl-by-month-parser.ts:126-134`) uses Xero's section title — that's the correct architecture. Don't replace with account-name pattern matching.
- **D-44.2-12:** Investigate the parent-section carry-forward logic at `pl-by-month-parser.ts:207-220`. Hypothesis: nested custom Xero sections (e.g., "Cost of Sales > Software Development > PK Costs" as nested section nodes) cause the inner section title to overwrite `currentParentTitle`, breaking inheritance.
- **D-44.2-13:** Test `standardLayout=true` vs `standardLayout=false` for JDS — capture what each returns. If `standardLayout=true` returns a more deterministic tree (consistent section names regardless of company customization), prefer it. If it loses fidelity (e.g., flattens custom accounts), keep `standardLayout=false` and fix the parser.
- **D-44.2-14:** Parent-title inheritance must NOT be broken by nested sub-sections. Fix: only update `currentParentTitle` when the section's title classifies to a known top-level type (revenue/cogs/opex/other_income/other_expense). Sub-sections like "Software Development" don't classify, so they don't clobber.

### Continuous integrity (post-fix)
- **D-44.2-15:** Daily reconciliation cron (separate from sync). Pulls Xero's P&L report for each `(business_id, tenant_id, fiscal_year=current)` and compares against `xero_pl_lines` aggregates. Failures mark the data_quality flag and Sentry-tag.
- **D-44.2-16:** This is BELT and BRACES vs the on-sync reconciler — sync-time reconciliation runs only when sync runs (could be hours apart from data drift); the daily cron catches drift between syncs.

### Scope discipline
- **D-44.2-17:** Out of scope: changing the Xero API endpoint, adding new sync sources (no Stripe / banking integration), changing the dual-business-ID system, modifying the wide→long migration, modifying the partial unique index from Phase 44.1, modifying the wizard's UI shell beyond adding the banner.
- **D-44.2-18:** In scope: parser fixes, sync_jobs schema additions, ForecastReadService data_quality field, banner wiring across 4 consumer routes, the daily cron, fixture-based regression tests, JDS + Envisage re-sync verification.

### Backfill / re-sync after fix
- **D-44.2-19:** After parser/classification fixes ship, re-sync ALL connected tenants (cron triggers full re-sync). Goal: every tenant's `sync_jobs.status = 'success'` post-fix. Any tenant that still fails is either (a) a Xero data quality issue at their end (rare), (b) a tenant-specific parser case we missed (likely; will need extension).
- **D-44.2-20:** No data deletion. The existing 1830 JDS rows in `xero_pl_lines` may be wrong-but-displayed. Re-sync upserts via the new partial unique index from Phase 44.1; rows with corrected `account_type` will be UPDATEd in place. Manual journal entries (`is_manual=true` in forecast_pl_lines, n/a here for xero_pl_lines) preserved.

### Hard constraints (locked from user 2026-04-28)
- Do NOT present false data to coaches. Period.
- Multi-tenant ("consolidated entities") MUST be first-class — per-tenant reconciliation, never aggregate-only.
- Accrual basis (already configured via `paymentsOnly=false` in sync orchestrator).
- This phase ships via PR (branch protection enabling per Phase 44q work) — no direct push to main.

### Claude's Discretion
- Banner UX language ("Data verification in progress — last sync failed against Xero" vs alternatives).
- Exact `data_quality` enum values + display logic.
- Fixture file format for raw Xero responses.
- Whether the daily cron runs at 02:00 AEST or another time.
- Tracking-category handling — diagnose and decide based on what the JDS fixture reveals.

<specifics>
## Specific Ideas

### data_quality enum (sketch)
```typescript
type DataQuality =
  | 'verified'    // most recent sync for tenant: status='success'
  | 'partial'     // most recent sync: status='partial' (reconciler caught discrepancies)
  | 'failed'      // most recent sync: status='failed' (sync threw, no data)
  | 'no_sync'     // no sync_jobs row for this (business_id, tenant_id)
  | 'stale'       // last successful sync >24h ago + no newer attempt
```
Banner copy / behavior matrix decided in plan.

### Banner contract (sketch)
```tsx
{dataQuality !== 'verified' && (
  <DataIntegrityBanner
    quality={dataQuality}
    lastSyncAt={lastSyncAt}
    discrepancyCount={...}
    onResync={() => triggerSync()}
    onViewDetail={() => openDiscrepancyDrawer()}
  />
)}
```

### Sync_jobs schema addition (sketch)
```sql
-- Idempotent: column add for tenant_id (currently nullable due to bug)
ALTER TABLE sync_jobs ALTER COLUMN tenant_id SET NOT NULL;
-- Backfill clean (nuclear) — drop existing rows since they have no tenant attribution
-- OR backfill from xero_connections lookup. Decide in plan.
```

### Reconciler current behavior (verified last night)
- Returns `{ status: 'success' | 'mismatch', discrepancies: [{account_code, account_name, fy_total, monthly_sum, diff}] }`.
- Tolerance is $0.01.
- For JDS: 66 accounts mismatched today, total absolute discrepancy ~$360K. Largest individual: Sales - Hardware $6,839.

## Deferred Ideas

- Cross-platform reconciliation (e.g., MYOB, QuickBooks) — same architectural pattern but separate implementation; revisit when those integrations exist.
- Self-healing sync (auto-retry mismatches) — too clever; manual coach intervention is honest.
- Real-time webhook from Xero on data change — Xero supports it; out of scope for this phase but a natural future direction.
- Discrepancy auto-categorization (e.g., "this $6,839 is a year-end journal") — could become useful with telemetry over time, but this phase establishes the gate and audit, not the smarts.
- Branch protection enabling — separately tracked; managed via Phase 44q (Codespaces) work.
