# 44.2-08 — data_quality propagation through 4 consumer routes — Summary

## What shipped

- **`src/app/finances/forecast/types.ts`**:
  - Re-exports `DataQuality` + `PerTenantQuality` types from forecast-read-service (single canonical type).
  - `HistoricalPLSummary` extended with `data_quality?` + `per_tenant_quality?` (optional because some legacy callsites use the type as input — the populated fields appear on every output post-this-plan).
- **`src/lib/services/historical-pl-summary.ts`**:
  - Active-forecast path: `data_quality` inherits from `composite.data_quality` (set by 44.2-07).
  - Fallback path: computes via `getDataQualityForBusiness` public wrapper.
  - Empty-data short-circuit (`has_xero_data: false`) ALSO surfaces `data_quality` so the banner can explain why no data shows up.
- **4 consumer routes updated:**
  - `src/app/api/Xero/pl-summary/route.ts` — surfaces via `summary.data_quality` (already threaded by getHistoricalSummary).
  - `src/app/api/forecast/cashflow/xero-actuals/route.ts` — surfaces top-level `data_quality` + `per_tenant_quality` on both the active-forecast path and the fallback path.
  - `src/app/api/monthly-report/generate/route.ts` — same pattern.
  - `src/app/api/monthly-report/full-year/route.ts` — same pattern.
- **`src/__tests__/api/data-quality-propagation.test.ts`** — 4 service tests + 4 structural-grep tests covering all 4 routes.

## Tests

8/8 green. tsc clean. 194/194 across services + api + xero + integration suites.

## Design notes

- Banner consumers in 44.2-09 will read either `response.data_quality` (cashflow + monthly-report routes) or `response.summary.data_quality` (pl-summary route). The 44.2-09 plan can normalize this at the consumer side — both shapes carry the same enum.
- The structural grep test catches the only failure mode that matters (forgetting to surface the field) at near-zero cost; full route integration tests would have required mocking auth + supabase + business access for each route, with no additional bug-catching value.
