# Phase 44.2 — Restructure Note

**Date:** 2026-04-30
**Author:** Matt Malouf + AI assist
**Trigger:** Empirical test of single-period vs by-month Xero query (2026-04-29) revealed a structurally broken aggregation API.

## Why this note exists

If you're reading this in 2027 and wondering why the directory has plans 01–03, then jumps to 06A–06F, then resumes at 07 — this is the explanation. Plans 04, 05, and the old 06 were **obsoleted mid-phase** because diagnosis on JDS revealed the original strategy was bandaging at the wrong layer.

## What changed

**Original plan (Wave 1):** treat the JDS Sales-Hardware $6,839 gap as either a parser bug (D-44.2-09) or a Xero API quirk requiring an absorber row (D-44.2-08). Plan 03 diagnosed the gap; plan 06 implemented the absorber.

**Reality (uncovered 2026-04-29):** the JDS gap is one of 35 systematically-negative gaps totaling $124,357 absolute, AND those gaps live SPECIFICALLY in the by-month aggregation API, not in our parser. Empirical proof:

- Calling `Reports/ProfitAndLoss?fromDate=2025-07-01&toDate=2025-07-31` (single-period for July 2025 only) returns Sales-Hardware = **$259,550.88** (matches Xero web PDF exactly).
- Calling `Reports/ProfitAndLoss?fromDate=2026-04-01&toDate=2026-04-30&periods=11&timeframe=MONTH` (the by-month query) returns Sales-Hardware July column = **$252,711.48** (off by exactly $6,839.40).

Same Xero data, two different query shapes, two different answers. Xero's own developer documentation acknowledges the bug:

> "If you use the periods parameter in combination with the fromDate and toDate parameters then the specified date range will apply to each period (i.e. if the specified date range is for a 30 day month, each prior period will only include the first 30 days)."

The by-month aggregator applies the same date-range length backwards N times instead of using calendar months. February (28 days) gets a 30-day window; the spillover days corrupt boundary months silently. This isn't a parser bug we can fix — it's documented Xero API behavior we have to design around.

## What replaced them

Six new plans encoding **Path A: per-month single-period queries** for both P&L and BS, with full Calxa-grade reconciliation:

- **44.2-06A** — P&L schema rebuild. Adds `account_id uuid` (Xero AccountID GUID, first-class), repurposes `account_code` to hold the user-facing Xero Code (200/300/etc.), adds `basis` stamp ('accruals'|'cash'), enforces `business_id` FK to `business_profiles(id)`. Includes one-shot backfill script that pulls `/api.xro/2.0/Accounts` per tenant and populates the catalog.
- **44.2-06B** — P&L sync orchestrator rebuild. Replaces `?periods=N&timeframe=MONTH` with N individual single-period calls. Folds in: classification fix (D-44.2-14), `FXGROUPID` row handling (multi-currency orgs), `business_profiles.fiscal_year_start` reading (no more hardcoded 7), `Organisation.Timezone` fetch, Calxa-spec rate-limit handler with `Retry-After` + `X-Rate-Limit-Problem` header support, sequential per-tenant fetches with 3-retry exponential backoff. Keeps `augmentWithResiduals` as a zero-adjustment regression detector.
- **44.2-06C** — Balance Sheet schema. Mirrors P&L design: `xero_bs_lines` with `account_id`, `account_code`, `basis`, FK, RLS. `xero_bs_lines_wide_compat` view. Point-in-time keyed (`balance_date`) instead of period-range.
- **44.2-06D** — Balance Sheet sync orchestrator. `Reports/BalanceSheet?date=YYYY-MM-LAST` per month-end. Net Assets == Equity runtime invariant.
- **44.2-06E** — 5-gate reconciliation harness. From Calxa-via-Cowork research:
    1. Σ(monthly PL) == FY-total PL
    2. PL ↔ BS articulation (Δ Current Year Earnings == PL net profit)
    3. TrialBalance balanced (Σ debits == Σ credits)
    4. BS in balance (Net Assets == Equity)
    5. Web parity (manual evidence vs Xero PDFs)
   Run automated for 3 tenants × 3 months = 36 assertions per CI commit.
- **44.2-06F** — Production migration runbook + rollback protocol + `verify-production-migration.ts`. Per-tenant canary cutover (JDS → Envisage → IICT). Feature-flag-gated for instant revert.

## Calxa intelligence that informed the restructure

Three findings from Calxa-via-Cowork research (full list in earlier conversation, summarized here):

1. **Xero's by-month API bug is a documented behavior, not a quirk.** Single-period queries are the only safe shape for both PL and BS.
2. **`Reports/ProfitAndLoss` + `Reports/BalanceSheet` are sufficient** for org-level reporting. No `Journals` API or full TrialBalance reconciliation pass needed in the production data path. (Calxa needs both because they rebuild from `Journals`; we don't.)
3. **TrialBalance is valuable as a verification gate**, not a production data source. The 5-gate harness in 06E uses TrialBalance for assertion 3 (Σ debits == Σ credits) — that's the right place for it.
