## Decisions Made

- **Outer RPC + inner direct writes coexist** rather than replacing begin/finalize with per-tenant inserts everywhere. The outer business-level row provides the D-07 single-flight guarantee that prevents two concurrent sync runs from racing; deleting it would regress an invariant Phase 44 Sub-A explicitly added. Per-tenant rows are layered ON TOP via direct `from('sync_jobs').insert/update` because their purpose is audit, not concurrency control.
- **Discrepancy tenant tagging at the wrapper, not the entry.** `reconciliation: { tenant_id, status, discrepant_accounts }` keeps payload size bounded even when a tenant has 60+ mismatching accounts (JDS today). Per-discrepancy `tenant_id` would multiply payload size N-fold for no audit gain.
- **Bounded error message length (500 chars).** Xero error stacks can be multi-KB; `.slice(0, 500)` at the orchestrator boundary keeps the `sync_jobs.error` column readable in Supabase Studio without a CHECK constraint. Operators get the head of the trace; full detail goes to Sentry via the per-tenant capture.
- **W3 contract change for two existing tests.** `finalize on thrown error` and `natural key uniqueness` previously asserted the orchestrator rethrew per-tenant errors. The new W3 contract is structurally incompatible with that — the whole point is that one tenant's failure does not propagate. Updated assertions: SyncResult.status='error' + outer finalize_xero_sync_job p_status='error' + per-tenant sync_jobs UPDATE carries the original error message. This was deliberate per the plan's W3 specification, not a regression.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] SyncResult missing `error` field when all tenants errored**

- **Found during:** Task 2 GREEN re-run (1/30 test failure: `finalize on thrown error` asserted `result.error` matched the failure regex but received empty string).
- **Issue:** When per-tenant try/catch absorbed all tenants' errors, the orchestrator computed `finalError = "All N tenants errored"` correctly but the SyncResult return shape didn't include the `error` field — it was only in the SyncResult literal returned by the outer-throw path further down.
- **Fix:** Added `...(finalError ? { error: finalError } : {})` spread to the success-path SyncResult so structured per-tenant aggregation messages reach callers.
- **Files modified:** `src/lib/xero/sync-orchestrator.ts`.
- **Committed in:** `d677a53` (Task 2 GREEN commit, pre-commit fix).

**2. [Rule 3 — Blocking] Existing `sync-orchestrator.test.ts` supabase mock didn't support `.insert()` / `.update()` chains**

- **Found during:** Task 2 first GREEN run (8/10 existing tests failed with `supabase.from(...).insert is not a function`).
- **Issue:** The 44-04 mock was designed for upsert-only on xero_pl_lines plus RPC calls; my orchestrator change introduces direct INSERT/UPDATE on sync_jobs which the mock didn't fluently support.
- **Fix:** Extended the mock to recognize `sync_jobs` table specifically: `.insert(payload).select('id').single()` returns a synthetic `{ id: 'tenant-job-N' }`; `.update(payload).eq('id', val)` resolves to a Promise and logs the payload + filter. Pattern is symmetric with the new test file's mock so the two test files use the same shape.
- **Files modified:** `src/__tests__/xero/sync-orchestrator.test.ts`.
- **Committed in:** `d677a53` (Task 2 GREEN commit).

**3. [Rule 2 — Critical] Two existing tests asserted the OLD throw-on-tenant-error contract**

- **Found during:** Task 2 first GREEN run (`finalize on thrown error` and `natural key uniqueness` still failed after mock fix).
- **Issue:** Both tests asserted `await expect(syncBusinessXeroPL(...)).rejects.toThrow(...)`. The W3 contract from the plan changes this — per-tenant errors do NOT propagate; the orchestrator returns a structured error result instead.
- **Fix:** Rewrote both assertions to verify the new contract: `result.status === 'error'`, per-tenant sync_jobs UPDATE carries the error text, outer finalize_xero_sync_job receives p_status='error'. Updated test docstrings to explicitly cite Phase 44.2-02 (W3) so future readers understand the contract change.
- **Files modified:** `src/__tests__/xero/sync-orchestrator.test.ts`.
- **Committed in:** `d677a53`.

### Architectural Rule-4 Considerations (resolved without checkpoint)

The plan's Task 2 example code uses direct `from('sync_jobs').insert / update` everywhere and removes the outer begin/finalize RPCs. The actual codebase uses begin_xero_sync_job (a critical D-07 single-flight invariant from Phase 44-05). Replacing it would have been a structural regression of an invariant the user explicitly locked in 44-05.

Resolved by **layering** rather than replacement: the outer RPC stays (claims tenant_id='' via migration DEFAULT — passes NOT NULL), and per-tenant rows go direct-insert inside the loop. The plan's acceptance criteria (`from('sync_jobs') >= 3`) is met (3 occurrences: 1 INSERT, 1 happy/partial UPDATE, 1 error UPDATE). All 5 plan-specified test cases pass against this architecture.
