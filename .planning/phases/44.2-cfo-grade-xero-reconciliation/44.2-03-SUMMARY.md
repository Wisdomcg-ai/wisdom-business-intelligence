# Phase 44.2 Plan 03: JDS Sales-Hardware Gap Diagnosis Summary

Cross-fixture diagnostic script + written diagnosis prove the $6,839.40 JDS Sales-Hardware gap (and 34 other JDS account gaps totalling $124,357) is a Xero API quirk, not a parser bug — directing 44.2-06 to implement an adjustment-row absorber rather than a parser fix.

## What was built

### 1. `scripts/diag-jds-sales-hardware-gap.ts` (421 lines)

Reusable, fixture-driven diagnostic that walks both `fy_query.response` and `by_month_query.response` from the JDS fixture captured in 44.2-01, joins per-account by Xero AccountID (with name fallback), and emits 4 sections of evidence:

- **Section A — Sales - Hardware deep dive:** account_id match check, full 12 monthly cells, FY-window sum, full sum, and signed diff lines.
- **Section B — All accounts with `|diff_fy26| > 0.01`:** ranked-by-magnitude table covering 35 accounts, with sign distribution and section concentration.
- **Section C — Orphan accounts:** 0 in FY-only, 5 in by-month-only (all FY25 transactions outside the FY26 query window, explained).
- **Section D — Hypothesis evidence summary:** sign pattern verdict (35-of-35 negative), top-5 sections by absolute diff, orphan-account explanation, Income-section row-diff sum vs Sales-Hardware diff (proves Sales-Hardware is the entire Income gap).

Pure offline tool (no DB / API calls); imports the fixture JSON directly. Takes <1s. Filters Xero summary rows (Gross Profit, Net Profit, Total *) so only real chart-of-account rows appear.

### 2. `.planning/phases/44.2-cfo-grade-xero-reconciliation/SALES-HARDWARE-GAP-DIAGNOSIS.md` (199 lines)

Written diagnosis embedding script output verbatim, evaluating six hypotheses (H1-H4 from D-44.2-07's probability list, plus H5 parser-bug, plus H6 NEW Xero-API-quirk), and arriving at the explicit Decision: **Path B — D-44.2-08 absorber**.

Cross-account scaling check confirms the absorber generalises across all 35 mismatched accounts (Sales - Hardware is the smallest of the larger gaps; ES - Installation Costs is the largest at -$31,987). Also addresses the FX revaluation sign-flip case ("Foreign Currency Gains and Losses": FY +$2,857 vs by-month -$2,196) and the FY25-orphan filtering for the 5 by-month-only accounts.

## Decisions Made

1. **Path B (absorber) over Path A (parser fix).** Decisive evidence: 35-of-35 negative sign pattern is structurally impossible from a random parser bug; both parsers walk Cells[] symmetrically; the diagnostic script reproduces the gap without using either parser.
2. **Absorber semantics:** synthetic `ParsedPLRow` per gap-bearing account, `amount = fy_total - sum(monthly_fy26)` (signed), `period_month = tenant FY-end month`, `notes` JSONB tag `{source: 'by_month_fy_gap_absorber', fy_total, monthly_sum, captured_from: 'fy_query'}` so audits can identify synthetic rows.
3. **Skip FY25 orphans:** the 5 by-month-only accounts have FY-window sum = $0 (non-zero values entirely in May/Jun 2025 outside the FY query window). 44.2-06 must filter these to avoid storing $0 rows.
4. **Dynamic FY-end month per tenant:** absorber must use `business_profiles.fiscal_year_end` not hard-coded `'30 Jun'`. Generality across tenants is required.
5. **Reconciler contract:** synthetic rows are zero-tolerance match (the gap is now in the data, not the discrepancy report). Sync orchestrator MUST log `"Absorbed N gap rows totalling $X across M accounts"` per tenant in `sync_jobs.reconciliation` JSONB.
