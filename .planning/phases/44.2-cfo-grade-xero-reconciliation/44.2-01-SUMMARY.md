## Decisions Made

- **Path A — Backfill-then-NOT-NULL** for the sync_jobs.tenant_id migration. Audit found 20/20 NULL rows are backfill-able from single-tenant `xero_connections` (JDS→Aeris, Envisage→Malouf Family Trust); 0 ambiguous, 0 prune-able. Rationale: every NULL row maps unambiguously to a single tenant; empty-string default (Path C) would discard recoverable audit history; pruning (Path B) would discard the 22:46:53 sync that surfaced the $359K reconciliation gap driving Phase 44.2.
- **Dual-ID bridge in migration SQL.** Both stored `sync_jobs.business_id` values are `business_profiles.id` (not `businesses.id`), so the migration MUST resolve via a `business_profiles` LEFT JOIN. Documented verbatim in the audit decision.
- **No `xc.is_active = true` filter in backfill SQL.** `xero_connections.is_active` flips transiently during sync runs (observed today on both JDS and Envisage); the row's existence is the authoritative signal for tenant attribution.

## Issues Encountered

- **`xero_connections.is_active` semantics are non-obvious.** The orchestrator at `src/lib/xero/sync-orchestrator.ts:325` filters `is_active=true` to pick which connections to sync, but elsewhere the column is flipped to false at the end of (or partway through) a sync run. This is independent of token validity (tokens are still refresh-able). Worth understanding before plan 44.2-02 ships its orchestrator changes — adding the per-tenant `sync_jobs` write needs to operate on the connection list as known at sync start, not at sync end.
- **Encryption key fallback.** `src/lib/utils/encryption.ts` falls back from `APP_SECRET_KEY` → `ENCRYPTION_KEY` → `SUPABASE_SERVICE_KEY` (deriving via PBKDF2). The Xero token decrypt worked from `SUPABASE_SERVICE_KEY` alone — no `ENCRYPTION_KEY` is set in the local env. Documenting this here so a later plan can decide whether to make the encryption-key-name explicit.
