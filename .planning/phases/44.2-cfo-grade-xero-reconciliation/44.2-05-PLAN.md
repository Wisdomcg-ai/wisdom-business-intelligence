<objective>
Fix the parent-section carry-forward bug per D-44.2-12 and D-44.2-14. JDS's "Software Development - PK Costs" is correctly Cost of Sales in Xero but classified as opex by our parser. Root cause: nested custom Xero sections like "Cost of Sales > Software Development > PK Costs" have an inner section title ("Software Development") that doesn't classify to a known top-level type, but the current parser still updates `currentParentTitle` to that inner non-classifying title — clobbering the inheritance from "Cost of Sales".

D-44.2-11: Don't replace section-title classification with account-name pattern matching. The architecture is correct; the inheritance gating is broken.

D-44.2-14: Only update `currentParentTitle` when the section's own title classifies to a known top-level type (revenue/cogs/opex/other_income/other_expense). Sub-sections like "Software Development" don't classify, so they don't clobber.

Output: parser fix + regression tests against fixtures.

```typescript
export function classifyAccountType(sectionTitle: string): AccountType {
  const t = sectionTitle.toLowerCase()
  if (t.includes('other income')) return 'other_income'
  if (t.includes('other expense')) return 'other_expense'
  if (t.includes('cost of sales') || t.includes('cogs') || t.includes('direct cost')) return 'cogs'
  if (t.includes('operating expense') || t.includes('expense')) return 'opex'
  if (t.includes('income') || t.includes('revenue') || t.includes('sales')) return 'revenue'
  return 'opex' // default for unknown sub-sections
}
```

    ```typescript
     * Strict classifier: returns null when the title does NOT match any
     * known top-level section type. Used by parsePLByMonth to gate the
     * parent-section carry-forward (D-44.2-14) — sub-sections like
     * "Software Development" must not clobber the inheritance chain.
     */
    export function classifyAccountTypeStrict(sectionTitle: string): AccountType | null {
      const t = sectionTitle.toLowerCase()
      if (t.includes('other income')) return 'other_income'
      if (t.includes('other expense')) return 'other_expense'
      if (t.includes('cost of sales') || t.includes('cogs') || t.includes('direct cost')) return 'cogs'
      if (t.includes('operating expense')) return 'opex'  // NOT generic 'expense' — strict requires the explicit phrase
      if (t.includes('income') || t.includes('revenue') || t.includes('sales')) return 'revenue'
      return null
    }
    ```

    Note the deliberate strictness on 'expense': the lenient classifyAccountType matches generic 'expense' (catching "Other Expenses" etc.); the strict helper requires the canonical "operating expense" phrase. This prevents random "Travel Expenses" sub-sections from looking like top-level OpEx anchors.
