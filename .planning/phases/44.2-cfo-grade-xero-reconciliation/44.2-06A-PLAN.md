<objective>
Path A foundation — schema rebuild for xero_pl_lines so account_id (Xero AccountID GUID) is first-class, account_code holds the actual user-facing Xero Code (sourced from /Accounts catalog), basis is stamped, business_id is FK-enforced. This is the prerequisite for 44.2-06B (sync orchestrator rewrite using per-month single-period queries).

Today the column NAMED `account_code` actually stores the AccountID GUID — a misleading shadow that has worked because every consumer treats it as an opaque key. Path A makes AccountID the canonical identity (it never changes when accountants rename or recode an account in Xero) and frees up `account_code` to hold the real Xero Code (200, 300, 400 — the human-facing chart-of-accounts code that coaches actually see in Xero's web UI).

Per the Phase 1 investigation, this also closes three structural fragilities that have been deferred:
- Q7: business_id FK enforcement (currently no FK, dual-ID system relies on convention)
- Calxa cash-vs-accrual best practice: stamp every cached row with its basis, never compare across baseses
- Q11 / Calxa best practice: pull /Accounts catalog per sync, key by AccountID GUID

This plan ships:
- 4 idempotent migrations (additive → backfill → constraint cutover → view update)
- 1 backfill script (pulls /Accounts per tenant, upserts xero_accounts, repopulates account_code from the catalog)
- 1 verification test suite asserting post-migration data integrity invariants

This plan does NOT change the sync orchestrator behavior — that is 44.2-06B's scope. After 06A merges, the orchestrator continues to write account_id (was: account_code holding GUID) into the new account_id column transparently. New sync writes will populate account_code from the xero_accounts catalog at sync time (06B work) — but for the bridge period between 06A merge and 06B merge, the backfill script's UPDATE of account_code is the only source of truth for that field, and any new sync between the two will leave account_code stale. This is acceptable because the bridge window is short and account_code is informational only (no consumer relies on it being the user-facing Code today; that's why repurposing is safe).

Per Q4 dual-key transition: account_id is NOT NULL after this plan. account_code remains NULLable to allow new syncs that haven't yet been wired to populate it. The constraint cutover migration (000003) drops the old (..., account_code, period_month) natural key and creates the new (..., account_id, period_month) natural key, atomic in one transaction.
</objective>
