<objective>
Path A core: replace the structurally-broken by-month query (`?periods=N&timeframe=MONTH` — Xero's documented date-arithmetic-applied-backwards bug) with N per-month single-period queries (`fromDate=YYYY-MM-01&toDate=YYYY-MM-LAST`), each returning the canonical Xero web report value for that single month.

Per the 2026-04-29 empirical proof: single-period Jul 2025 query returns Sales-Hardware $259,550.88 — exactly matches Xero web PDF. By-month query says $252,711.48 — off by exactly $6,839.40 (the JDS smoking-gun gap). Same pattern across 35 JDS accounts and the 5 orphan accounts. Path A is the only sufficient fix.

This plan also folds in everything that should have been done years ago and was deferred:
- Classification fix (D-44.2-14) — parent-section carry-forward bug; required for "Software Development - PK Costs" misclassified as opex (should be cogs). Was supposed to be 44.2-05; folded in here.
- FXGROUPID handling — Xero docs: "the standard layout of the profit and loss report will group multi currency system accounts into a single line with a Value of FXGROUPID (instead of an AccountID). This is not the case for US organisations and Australian demo companies." For multi-currency orgs (IICT HK subsidiary in scope), this row was silently corrupted by the parser today.
- fiscal_year_start from business_profiles — currently hardcoded `7` (Australian July); breaks for non-AU tenants
- Organisation.Timezone fetch — eliminates UTC-vs-org-local FY-rollover boundary hazard
- /Accounts catalog refresh — Calxa-best-practice, gives user-facing Xero Code (200/300/400) for every account
- Rate-limit handler — concrete spec from Xero docs; production-ready 429 + Retry-After + 5xx behavior
- account_id GUID first-class write (using 06A's new column)
- basis='accruals' stamp on every row (using 06A's new column)
- augmentWithResiduals retained as regression detector (zero adjustments expected; non-zero = alert)

This plan does NOT change xero_pl_lines schema (06A did that) or add Balance Sheet sync (06D will). It is the single largest behavioral change in Phase 44.2 — and the one that finally makes JDS YTD-Mar match the Xero PDF to the cent.

Output: orchestrator rewrite + 5 new modules + 3 new test files + 2 updated test files. Two existing PRs (#22, #23) are preserved unchanged; this builds on top.
</objective>

  Concurrent: 5 in-flight

  X-DayLimit-Remaining, X-MinLimit-Remaining, X-AppMinLimit-Remaining
