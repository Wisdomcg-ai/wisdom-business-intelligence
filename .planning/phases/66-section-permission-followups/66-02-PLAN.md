<objective>
Normalize business-ID resolution across the three consolidated routes (`consolidated/`, `consolidated-bs/`, `consolidated-cashflow/`) so they call `resolveBusinessIds` at the top of the handler and use the resolved `businesses.id` for the access check, the section-permission gate, the `business_profiles` lookup, and the engine call (D-06).

Purpose: All other Phase 65-02 finance routes already resolve dual IDs; these three pass `business_id` from the request body straight through. The consolidation engine resolves IDs internally so data fetches are safe, but the route-level access check (`.eq('id', business_id)`) and `requireSectionPermission` both assume `business_id` is a `businesses.id`. A caller passing a `business_profiles.id` would silently fail the access check before the section gate runs. Live risk is LOW (the frontend sends `businesses.id`) — this is defensive normalization for consistency and safety.

Output: Three route files updated to resolve IDs once and use `ids.bizId` consistently, plus one regression test.
</objective>

  export async function resolveBusinessIds(
    supabase: { from: (table: string) => any },
    businessId: string,

  // ids.profileId — use for business_profiles, xero_connections, financial_forecasts

  // Has a module-level Map cache; repeated calls within a request are free.
