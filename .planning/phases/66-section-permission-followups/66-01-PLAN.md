The migration backfills the canonical `finances` key into `section_permissions` for rows that only carry the legacy `financials` key (or neither key). Wrap in a transaction and make it idempotent via the `WHERE NOT (section_permissions ? 'finances')` clause.

Header comment: explain the legacy/canonical drift (baseline DEFAULT uses `financials`, Phase 65 `requireSectionPermission` reads `finances`), state that this is the Phase 65 Wave 65-04 prerequisite, and note the backfill rule.

Body — exactly two UPDATE statements inside one transaction:

```sql
BEGIN;

-- business_users: backfill the canonical 'finances' key for rows that only
-- carry the legacy 'financials' key (or neither key).
-- Rule: finances <- financials value when present; <- true when neither key
-- exists (matches the original baseline DEFAULT intent — least-surprise:
-- retroactively allow rather than deny someone who never had an explicit deny).
UPDATE public.business_users
SET section_permissions = section_permissions || jsonb_build_object(
  'finances',
  COALESCE((section_permissions->>'financials')::boolean, true)
)
WHERE NOT (section_permissions ? 'finances');

-- team_invites: same backfill for the staging table. Impact is lower because
-- invite acceptance writes section_permissions from the request body, not by
-- copying this column (team/invite/route.ts uses `sectionPermissions || {}`).
-- Included for cleanliness / future-proofing the stale column DEFAULT.
-- Note: the team_invites baseline DEFAULT for financials is false — confirm
-- the exact value at baseline line ~5206 before finalizing this comment.
UPDATE public.team_invites
SET section_permissions = section_permissions || jsonb_build_object(
  'finances',
  COALESCE((section_permissions->>'financials')::boolean, false)
)
WHERE NOT (section_permissions ? 'finances');

COMMIT;
```

CRITICAL constraints (research Pitfall 2):
- MUST use the `||` JSONB merge operator — `section_permissions || jsonb_build_object(...)`. NEVER assign a bare object (`section_permissions = '{"finances": ...}'::jsonb`), which would wipe every other key.
- The `WHERE NOT (section_permissions ? 'finances')` clause is what makes the migration idempotent — rows that already have `finances` are untouched, including any conflicting rows the audit flagged.
- Do NOT change the column DEFAULT and do NOT add/drop columns — this is a data backfill only, scoped strictly to the two named tables.
- Verify the exact `team_invites` baseline DEFAULT value at baseline_schema.sql line ~5206 and make the `COALESCE` fallback comment match it (research states `financials: false`).
