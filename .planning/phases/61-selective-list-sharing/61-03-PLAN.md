<objective>
Extend `dailyTasksService` and `ideasService` to make sharing reachable from application code, surface ownership to UI via `is_owner`, and patch the three pre-existing ownership gaps in `ideasService` on the way through.

Purpose: Without this layer, neither the API (61-04) nor the UI (61-05) has a clean entry point. This is where "what does the row owner see vs the recipient see" first becomes explicit.

Output: Two modified service modules + two paired vitest suites. No DB changes. No new API routes.
</objective>

```typescript
export async function getTodaysTasks(overrideUserId?: string): Promise<Task[]> {
  const supabase = createClient()
  const userId = await getEffectiveUserId(overrideUserId)
  if (!userId) return []
  const { data, error } = await supabase
    .from('daily_tasks')
    .select('*')
    .eq('user_id', userId)        // ← REMOVE: RLS already gates. Replace with broader fetch + is_owner mapping.
    .neq('status', 'done')
    .is('archived_at', null)
    .order('created_at', { ascending: false })
  if (error) { console.error(...); return [] }
  return data as Task[]
}
```

After this plan, the same function:

- maps result rows: `(data ?? []).map(r => ({ ...r, is_owner: r.user_id === userId }))`
- returns `(Task & { is_owner: boolean })[]`
