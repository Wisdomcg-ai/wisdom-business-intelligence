<objective>
Fix the converter root cause for D-44.1-06 vectors 2 and 3: `convertAssumptionsToPLLines` currently REPLACES `forecast_months` on every line it produces. When the input has empty `year1Monthly` (vector 2) or shrunk `forecastDuration` (vector 3), the resulting `forecast_months` JSONB blanks Year-1 keys / drops Y2-Y3 keys that previously existed on the matched DB row.

Per the locked decision (extending 44.1 scope, in-session 2026-04-29): the structural fix lives at the converter, not the RPC. JSONB-column-level loss is data loss; UPSERT alone doesn't fix it. The merge pattern below preserves existing keys while letting input keys win per-key — input is the source of truth for what it provides; existing is the source of truth for what it doesn't.

This plan is **autonomous: true** because it is a localized, mechanical refactor of 5 sub-converters in a pure-function file with no external IO. Tests in 44.1-03 (wave 3) will validate it end-to-end.

The converter's contract change:
- BEFORE: `forecast_months = { ...computedFromInput }` (replaces existing)
- AFTER:  `forecast_months = { ...existing.forecast_months, ...computedFromInput }` (merges; input wins per-key, unmentioned existing keys survive)

The 5 sub-converters and their current line ranges in assumptions-to-pl-lines.ts:
- `convertRevenue` — lines 128-188 (currently builds `forecastMonths: Record<string, number> = {}` at line 140)
- `convertCOGS` — lines 194-266 (currently builds `forecastMonths: Record<string, number> = {}` at line 207)
- `convertOpEx` — lines 272-378 (currently builds `forecastMonths: Record<string, number> = {}` at line 285)
- `convertTeam` — lines 384-567 (more complex: builds 4 separate per-month maps; merge applies to the FINAL `tl.data` per teamLine in the loop at lines 545-564)
- `convertCapExDepreciation` — lines 573-609 (currently builds `forecastMonths: Record<string, number> = {}` at line 590)

Pattern (literal, to be embedded in each function — adapt variable names per function):
```typescript
// Start the new forecast_months from the existing row's value (so unmentioned keys survive).
// Then overlay input-provided keys (input wins per-key).
const newForecastMonths: Record<string, number> = { ...(existing?.forecast_months || {}) }
// ... existing logic that previously assigned to forecastMonths now assigns to newForecastMonths
// when forecastMonthKeys.includes(mk). The "if (forecastMonthKeys.includes(mk))" guard is preserved
// to avoid leaking out-of-range months from input.
```

For `convertCapExDepreciation` and the per-month-loop branches in convertOpEx (fixed/variable/seasonal/adhoc), the pattern is the same: build `newForecastMonths` starting from existing, then assign per-key. Note: in those branches the loop iterates `forecastMonthKeys` and writes EVERY key — that is a full overwrite for in-range keys. That's correct: the merge only protects keys OUTSIDE `forecastMonthKeys` (e.g., Y2/Y3 keys when forecastDuration=1, or any key the input doesn't iterate over). Existing keys IN-range will be overwritten by the input's output, which is the right behavior.

For `convertTeam`, the existing code initialises 4 maps to 0 for every key in `forecastMonthKeys`, then accumulates. The merge applies at the BOTTOM of the function (lines 545-564) where `tl.data` becomes a row's `forecast_months`. There, change to:
```typescript
const merged: Record<string, number> = { ...(existing?.forecast_months || {}), ...tl.data }
```
This way: any month in `forecastMonthKeys` is overwritten by `tl.data` (correct — team aggregation is authoritative for in-range months); months OUTSIDE `forecastMonthKeys` (e.g., Y2/Y3 when forecastDuration=1) survive from the existing row.
