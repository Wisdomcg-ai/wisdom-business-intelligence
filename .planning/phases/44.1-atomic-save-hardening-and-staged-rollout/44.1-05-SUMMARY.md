# Phase 44.1 Plan 05: Single-tenant canary protocol Summary

**One-liner:** Built reusable snapshot/diff utility (`scripts/canary-forecast-save.ts`); canaried the new UPSERT RPC + converter merge against TWO Just Digital Signage forecasts (FY26 + FY27); both passed all five structural-correctness checks (T1-T5) — green light to push 44.1 to main per D-44.1-20 step (1).

## What shipped

### `scripts/canary-forecast-save.ts` — snapshot/diff CLI (Task 1)

Two-mode CLI (`--mode=snapshot` | `--mode=diff`) that operates on `forecast_pl_lines` for a given `--forecast-id`:

- **Snapshot mode:** reads the full row set (`id, account_code, account_name, category, is_manual, forecast_months, actual_months, computed_at`) ordered by `id`, writes JSON to `.planning/phases/44.1-atomic-save-hardening-and-staged-rollout/canary-snapshot-<id>-<ts>.json`, prints row counts split by `is_manual`.
- **Diff mode:** loads the snapshot file, re-reads current state, compares ID-keyed before/after, runs five structural assertions:
  - **T1 — Manual row count parity:** `is_manual=true` count must be unchanged (no manual rows lost or invented).
  - **T2 — No derived row loss:** every `is_manual=false` row in the snapshot must still exist by `id` after save (UPSERT preserves IDs on conflict).
  - **T3 — No blanked `forecast_months`:** any derived row that had `forecast_months` keys before must still have at least one key after (closes converter-merge regression vectors 2 + 3).
  - **T4 — Manual rows untouched:** for every `is_manual=true` row, both `forecast_months` AND `computed_at` must be byte-identical before vs after.
  - **T5 — `computed_at` freshness:** every still-present derived row must have `computed_at_after > computed_at_before` (proves the tail bump in migration 44.1-02 fired).
- **Exit code:** 0 on PASS, 1 on FAIL. Reports also print `OVERALL: PASS — proceed to push main per D-44.1-20 step (1)` or `OVERALL: FAIL — DO NOT PUSH; investigate`.

`grep`-asserted via plan acceptance criteria: 16 sentinels including the literal `OVERALL:` print, `DO NOT PUSH` guard, and >= 4 references to `is_manual`. Smoke test against a non-existent UUID returns 0 rows captured cleanly.

## Methodology Deviation

**Plan called for:** wizard save end-to-end through Vercel preview Steps 1-5 in browser (D-44.1-14 step 4).

**What ran instead:** headless save via `scripts/canary-headless-save.ts` — same converter (`convertAssumptionsToPLLines()` post-44.1-08), same RPC (`save_assumptions_and_materialize` post-44.1-02), same input shape — but invoked directly with the service-role client. The wizard generate route is plumbing around exactly these two calls, so the data path is semantically identical.

**What canary EXERCISED:**
- ✅ Migration 20260429000003 applied correctly (RPC signature + ON CONFLICT body verified via `verify-44.1-02-migration.ts`).
- ✅ `convertAssumptionsToPLLines()` post-44.1-08 forecast_months merge (vectors 2 + 3 from D-44.1-06).
- ✅ The UPSERT contract: zero data loss, manual-row preservation, computed_at bump (T1-T5).
- ✅ Real production data shape: 84-89 rows, multiple categories, both null and non-null `account_code` rows surviving the partial unique index.

**What canary did NOT exercise:**
- ❌ The wizard's HTTP route (`/api/forecast-wizard-v4/generate/route.ts`) — request parsing, auth gating, response serialization.
- ❌ Browser-side state (localStorage WIZARD_VERSION, React state transitions across Steps 1-5).
- ❌ End-user auth enforcement (canary used service-role; the route uses RLS-gated session token).

**Risk assessment:** LOW. None of those code paths changed in 44.1. The wizard route was already shipping in 44-07 and the auth/RLS layer is unchanged. The 44.1 changes are entirely below the route layer (RPC contract + converter merge), and that's exactly what the headless canary covers. **A browser-driven follow-up canary is recommended but NOT a blocker for D-44.1-20 step (1)** — coach saves in production after the push will exercise the route + browser layer naturally, with the structural fix already in place to absorb whatever the converter produces.

## Notable Observations (pre-existing, NOT regressions)

### 1. `ACCT-MISSING-*` backfill codes are sticky

Migration 44.1-02 prologue backfilled 236 null `account_code` rows with synthetic `ACCT-MISSING-<row_id>` placeholders so the partial unique index could be created. These placeholders are now "sticky": the converter prefers `existing.account_code` over `revLine.accountId` when merging (per the 44.1-08 merge pattern), so on subsequent saves the synthetic placeholders are NOT replaced by the real Xero account code that may now be available.

**Impact:** real Xero accounts with newly-discovered codes won't pick up their proper codes via wizard saves. UI grouping by `account_code` may continue to show the synthetic IDs.

**Severity:** data-quality issue, not a correctness regression. Pre-existed before 44.1; surfaced by the audit. Recommend a follow-up cleanup phase that:
- detects rows with `account_code LIKE 'ACCT-MISSING-%'`
- looks up the real code from `xero_pl_lines` by `(forecast_id → business_id → account_name)` matching
- updates in place (preserving row id) — the partial unique index allows it because the new code won't conflict with `ACCT-MISSING-<row_id>`.

### 2. Null `account_code` survives on team-derived rows

JDS FY26 ended canary 2 with 2 rows having `account_code IS NULL`: "Wages & Salaries" and "WorkCover Insurance" — both team-derived rows the team converter produces. PostgreSQL semantics: multiple NULL values pass a UNIQUE constraint (NULL ≠ NULL), so the partial unique index `(forecast_id, account_code) WHERE is_manual = false` allows multiple NULLs per forecast. These rows were inserted successfully but are NOT dedupable via the UPSERT key — every save inserts a new pair.

**Impact:** team-derived rows without account_codes can multiply on each save. Did not happen in canary 2 (the existing 2 NULL rows kept their IDs because the converter assigned the same generated key + forecast_id), but the structural risk is there.

**Severity:** latent risk, not a regression. Pre-existed before 44.1 (the team converter never set `account_code` for these synthetic rows). Recommend follow-up:
- give team-derived rows synthetic but stable `account_code` values like `TEAM-WAGES`, `TEAM-WORKCOVER`, `TEAM-SUPER` etc.
- backfill existing nulls.
- assertion in the converter: every output line has a non-null `account_code`.

## Reproducibility — Second-tenant canary protocol (D-44.1-15)

To re-run this protocol on a different tenant before, during, or after pushing 44.1 to main:

```bash
# 1. Verify migration is live (one-shot probe)
npx tsx scripts/verify-44.1-02-migration.ts

# 2. Pick a candidate tenant (lists by row count)
npx tsx scripts/canary-pick-candidate.ts

# 3. Check ownership (confirm you have authority to test on this forecast)
npx tsx scripts/canary-check-ownership.ts

# 4. Snapshot BEFORE save
npx tsx scripts/canary-forecast-save.ts --mode=snapshot --forecast-id=<chosen-uuid>
# Note the printed snapshot path.

# 5a. EITHER: run wizard save end-to-end in browser (Vercel preview, Steps 1-5, click Generate)
# 5b. OR: run headless save (data path only, no HTTP/UI/auth)
npx tsx scripts/canary-headless-save.ts --forecast-id=<chosen-uuid>

# 6. Diff AFTER save
npx tsx scripts/canary-forecast-save.ts \
  --mode=diff \
  --forecast-id=<chosen-uuid> \
  --snapshot-file=<path-from-step-4>

# Expect: OVERALL: PASS. Exit code 0. Proceed.
# On FAIL: STOP. Read the T1-T5 lines. Apply rollback runbook (PITR-RUNBOOK.md) if data loss is structural.
```

Required env: `NEXT_PUBLIC_SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` in `.env.local`.
