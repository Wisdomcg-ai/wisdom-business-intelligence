# Phase 44.1 Plan 04: Soft-fail invariant gate Summary

**One-liner:** `ForecastReadService` D-18 invariants gated behind `FORECAST_INVARIANTS_STRICT` env flag — default-soft logs to Sentry with tag `invariant_violation_logged` and returns the row; strict throws as before.

## What shipped

**`src/lib/services/forecast-read-service.ts`** — module-level constant `STRICT_INVARIANTS = process.env.FORECAST_INVARIANTS_STRICT === 'true'` (read once at module load). Both `assertComputedAtIsFresh` and `assertCoverageNonNegative` now branch on this constant:

| Mode | `STRICT_INVARIANTS` | Behavior on violation | Sentry call | Sentry tag |
|------|---------------------|-----------------------|-------------|----------|
| **Strict** (post-soak) | `true` | Throws Error → 500 to caller | `captureException` | `invariant: 'forecast_freshness'` or `'coverage_non_negative'` |
| **Soft** (default — soak) | unset / anything else | Returns the row; no throw | `captureMessage` (warning) + `addBreadcrumb` | `invariant_violation_logged: 'forecast_freshness'` or `'coverage_non_negative'` |

Soft-fail breadcrumb / extra payload includes `forecast_id`, `computed_at`, `assumptions_updated_at`, and a derived `delta_seconds` for triage, per D-44.1-09.

**`src/__tests__/services/forecast-read-service.test.ts`** — 7 existing tests (multi-tenant aggregate, parity, invariant, negative coverage, active-forecast, subtotals, cashflow) preserved. Added a `Soft-fail invariant mode — D-44.1-08` describe block with 4 new tests, all going through the public `getMonthlyComposite` surface:

1. `strict-off — stale computed_at logs to Sentry but does NOT throw via getMonthlyComposite (D-44.1-09)` — asserts `captureMessage` + `addBreadcrumb` fire with `invariant_violation_logged: 'forecast_freshness'`, AND that the call resolves with a defined row.
2. `strict-off — negative coverage logs to Sentry but does NOT throw via getMonthlyComposite (D-44.1-09, W3)` — same shape for the coverage invariant. Goes end-to-end through the public method (W3 — does NOT call private `assertCoverageNonNegative`).
3. `strict-on — negative coverage THROWS via getMonthlyComposite (sentinel)` — confirms the strict throw path remains intact end-to-end through the public method.
4. `strict-off — fresh computed_at does NOT log invariant_violation_logged (sentinel)` — no false positives when the invariants are satisfied.

The strict-on tests rely on `vi.hoisted(() => { process.env.FORECAST_INVARIANTS_STRICT = 'true' })` to set the env var BEFORE the module's top-level `STRICT_INVARIANTS` constant is evaluated. This was necessary because vitest hoists `vi.mock()` and ESM `import` statements above plain top-of-file statements; only `vi.hoisted` runs early enough.

**`.env.example`** — appended a Phase-44.1 documentation block (idempotent — `grep -c FORECAST_INVARIANTS_STRICT .env.example` returns exactly 1) with default `FORECAST_INVARIANTS_STRICT=false` and a comment cross-referencing 44.1-CONTEXT.md (D-44.1-08 through D-44.1-12).

## Soak protocol (D-44.1-11) — operator runbook

When the Wave 8/9 batch ships to main (D-44.1-20 step 3):

1. Production env var `FORECAST_INVARIANTS_STRICT` MUST be unset OR `false`. Verify in Vercel dashboard before merging.
2. Watch Sentry for events tagged `invariant_violation_logged: forecast_freshness` AND `invariant_violation_logged: coverage_non_negative` for 24-48 hours.
3. Expected: legacy serial-saved forecasts will surface here. Triage population: per-tenant counts, per-forecast counts. Run `POST /api/forecast/{id}/recompute` for any forecast that should be remediated.
4. Once Sentry shows zero new `invariant_violation_logged` events for the active-tenant cohort across a 24-48h window: flip Vercel env var `FORECAST_INVARIANTS_STRICT=true`. Trigger a redeploy so the new module-load value takes effect.
5. From that moment forward, any new violation throws + 500s + tags Sentry `invariant`. Strict mode is back to what Plan 44-08 originally shipped.

The flip is a Vercel env-var change, NOT a code change. No PR. No deploy. Just env var + redeploy.
