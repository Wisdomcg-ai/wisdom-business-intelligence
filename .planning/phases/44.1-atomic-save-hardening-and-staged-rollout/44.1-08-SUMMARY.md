# Phase 44.1 Plan 08: Converter Passthrough Fix — Merge Existing forecast_months Summary

**One-liner:** convertAssumptionsToPLLines now merges existing forecast_months per-key in all 5 sub-converters, structurally closing D-44.1-06 vectors 2 + 3 at the upstream root cause.

## Objective Recap

Vectors 2 and 3 of the D-44.1-06 multi-vector regression brief described JSONB-column-level data loss caused by the converter producing `forecast_months = { ...computedFromInput }`:

- **Vector 2:** When input has no `year1Monthly`, the existing row's Year-1 keys are blanked because the converter writes a fresh empty object.
- **Vector 3:** When `forecastDuration` shrinks (e.g., 3 → 1), Y2/Y3 month keys are dropped because the converter only writes keys for Year-1 months.

The fix replaces the contract:

- **Before:** `forecast_months = { ...computedFromInput }` (replaces existing — JSONB-level loss)
- **After:** `forecast_months = { ...existing.forecast_months, ...computedFromInput }` (merges; input wins per-key, unmentioned existing keys survive)

This is the structural fix per the locked decision in CONTEXT.md (in-session 2026-04-29): JSONB-column-level loss is real data loss, and an UPSERT alone (44.1-01) cannot fix it because UPSERT replaces the column wholesale. The merge must happen upstream at the converter.

## What Changed

Five edits to `src/app/finances/forecast/services/assumptions-to-pl-lines.ts`. Every PLLine produced by every sub-converter now starts its `forecast_months` from `{ ...(existing?.forecast_months || {}) }` and then overlays input-derived keys.

### Canonical merge pattern (Revenue, COGS, OpEx, CapEx Depreciation)

```typescript
// D-44.1-06 vector 2/3 — start from existing forecast_months so unmentioned keys survive.
const newForecastMonths: Record<string, number> = { ...(existing?.forecast_months || {}) }

// ... existing per-key writes (Y1, Y2 quarterly, Y3 quarterly, fixed, variable, seasonal, adhoc)
// now write to newForecastMonths instead of forecastMonths.
// The "if (forecastMonthKeys.includes(mk))" guard is preserved to avoid leaking
// out-of-range months from input.

lines.push({
  ...,
  forecast_months: newForecastMonths,
  ...,
})
```

### Team converter — slightly different shape

`convertTeam` pre-initialises 4 separate per-month maps (`wagesPerMonth`, `superPerMonth`, `workCoverPerMonth`, `payrollTaxPerMonth`) to 0 for every key in `forecastMonthKeys`, then accumulates contributions from `existingTeam`, `plannedHires`, `bonuses`, `commissions`, and on-cost calculations. The merge applies at the BOTTOM of the function — inside the `for (const tl of teamLines)` push loop where each `tl.data` becomes a row's `forecast_months`:

```typescript
// D-44.1-06 vector 2/3 — merge tl.data over existing forecast_months. tl.data covers
// every key in forecastMonthKeys (initialised to 0 then accumulated), so it overwrites
// in-range keys; out-of-range keys (e.g., Y2/Y3 when forecastDuration=1) survive from
// existing.
const newForecastMonths: Record<string, number> = { ...(existing?.forecast_months || {}), ...tl.data }
```

This is the right shape because Team-style aggregation is authoritative for every month it computes (the pre-init-to-zero step makes it produce a complete map across `forecastMonthKeys`). The merge only protects months OUTSIDE that range — exactly what vectors 2 + 3 require.

### CapEx Depreciation — `existing` lookup moved before the loop

```typescript
// Before: existing was looked up AFTER the loop populated forecastMonths. Order didn't matter.
// After: existing must be available BEFORE the loop so the seed object can read existing.forecast_months.

const existing = findMatchingLine(existingLines, 'Depreciation')

// D-44.1-06 vector 2/3 — start from existing so out-of-range keys survive.
const newForecastMonths: Record<string, number> = { ...(existing?.forecast_months || {}) }
for (const mk of forecastMonthKeys) {
  newForecastMonths[mk] = round2(totalMonthlyDepreciation)
}
```

## Behavior Matrix

| Scenario | Before this plan | After this plan |
|---|---|---|
| Input has full year1Monthly + Y2/Y3 quarterly, existing has same shape | Existing replaced — same final shape | Existing replaced — same final shape (no observable change) |
| Input has no year1Monthly (vector 2) | existing.forecast_months blanked → DATA LOSS | existing.forecast_months preserved verbatim |
| Input has year1Monthly but forecastDuration=1 (vector 3); existing has Y2/Y3 keys | Y2/Y3 keys DROPPED → DATA LOSS | Y2/Y3 keys preserved; Y1 in-range keys overwritten by input |
| No existing match (new account) | `forecastMonths = {}` then built from input | `newForecastMonths = { ...({}) } = {}` then built from input — SAME |
| Existing has key X (Y1, in-range) and input writes X | Input wins | Input wins (same; merge does NOT change in-range overwrite semantics) |
