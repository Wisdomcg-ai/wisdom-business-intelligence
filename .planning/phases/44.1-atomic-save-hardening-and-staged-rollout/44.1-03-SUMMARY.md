## Accomplishments

Six new tests added in a new `describe('Loss-vector regression — D-44.1-06', ...)` block at the end of the file. Each test names match VERBATIM per the plan:

1. **`vector 1 — empty opex lines preserves all 5 existing OpEx rows (D-44.1-06)`** — converter returns ONLY revenue (no opex); asserts all 5 OpEx rows survive untouched (UPSERT bypasses unmentioned account_codes).
2. **`vector 2 — undefined year1Monthly does not blank forecast_months on existing rows (D-44.1-06)`** — STRICT preservation. Existing `{ '2026-05-01': 100000, '2026-06-01': 110000 }`, input has empty `inputForecastMonths`. Asserts post-RPC the row's `forecast_months` is exactly equal to the original.
3. **`vector 3 — shrunk forecastDuration preserves Y2/Y3 month keys on existing rows (D-44.1-06)`** — STRICT preservation. Existing has 12 keys (Y1+Y2+Y3); input only writes Y1; asserts post-RPC all 12 keys present, Y1 overwritten with input values, Y2 + Y3 preserved with original values.
4. **`vector 4 — RLS-empty existingLines preserves is_manual=true rows (D-44.1-06)`** — RLS scenario; the manual row's `forecast_months` and `computed_at` are untouched while the derived row is upserted.
5. **`vector 5 — manual-flag duplicate does NOT create two rows for same (forecast_id, account_code) (D-44.1-06)`** — DB has manual row at account_code='4000'; input writes derived row at same code. Calls POST twice. Asserts exactly 1 manual + 1 derived row remain (no duplication on re-run).
6. **`vector 6 — force_full_replace=true clears non-manual rows AND preserves manual rows (W2)`** — 3 derived rows + 1 manual. Force-replace + 1 input row → 1 input row + 1 manual row remain (2 derived rows dropped, manual preserved).

## Strict preservation note (vectors 2 + 3)

Per the plan-checker B3 directive, vectors 2 and 3 assert preservation STRICTLY — no "documents current behavior" caveat language. This is structurally guaranteed by the chain:

- **44.1-08** patches the converter so `forecast_months = { ...existing.forecast_months, ...inputDerived }` in all 5 sub-converters. When input doesn't provide a key, the existing key survives.
- **44.1-02** ships the UPSERT RPC. The RPC's `DO UPDATE SET forecast_months = EXCLUDED.forecast_months` writes whatever the converter handed it — i.e., the merged JSONB.
- **44.1-03** locks both contracts: `makeConverterMockMerging` mirrors the 44.1-08 merge, and `applyUpsertToSimulatedDB` mirrors the 44.1-02 UPSERT body. Any regression in either layer fails these tests.

## force_full_replace=true (vector 6)

Vector 6 covers the W2 force-replace branch at the RPC contract level. The current route handler doesn't yet pass `p_force_full_replace` (the parameter defaults to `false`), so the test simulates the flag at the mock boundary by injecting `argsWithForce = { ...args, p_force_full_replace: true }` before calling `applyUpsertToSimulatedDB`. This is the test of the RPC contract: when called with `force_full_replace=true`, the partial-replace branch must:

- Delete all rows for the forecast where `is_manual = false`.
- Then UPSERT the input rows (which re-INSERT after the force-clear).
- Leave `is_manual = true` rows entirely untouched (no `computed_at` bump on manual rows).

## JS-side simulation of the contract

The substantive innovation here is `applyUpsertToSimulatedDB`: a 50-line JS function that mirrors the SQL body of `save_assumptions_and_materialize` (post-44.1-02). It:

1. Optionally executes the force-full-replace branch (delete is_manual=false rows for the forecast).
2. For each input line, finds a matching is_manual=false row by `(forecast_id, account_code)`. If found, UPDATE; else INSERT (always with `is_manual = false` hardcoded).
3. Bumps `computed_at` on un-touched is_manual=false rows to mirror the tail UPDATE in the SQL (Section 5 of the function body).

The mock's mutation of `simulatedDB` IS the contract pin. Tests fail if the route asks the RPC for a wrong shape OR if the simulation contradicts the SQL spec. Together with `makeConverterMockMerging` (mirroring the 44.1-08 merge), the JS layer is a faithful regression fence.

## Decisions Made

See frontmatter `key-decisions` for the full list. Key highlights:

- **Tests run at the route boundary** — mock supabase.rpc + the converter; verify what the route ASKS the RPC to write (call args + simulated post-state). This is more useful than unit-testing the SQL or the converter in isolation, because it catches drift across the whole call chain.
- **Mock APPLIES upsert semantics, not just records calls** — `applyUpsertToSimulatedDB` mutates the simulated DB exactly as the SQL would. If the route handler ever drifts (e.g., omits a field, double-calls, swaps args), the simulation produces a wrong post-state and the test fails.
- **Vectors 2 + 3 strict preservation is now structurally guaranteed**, not aspirational — the converter merges (44.1-08), the RPC writes the merged result (44.1-02), and these tests pin both.
