**Replaced the DELETE-then-INSERT body of `save_assumptions_and_materialize` with structural UPSERT semantics keyed on `(forecast_id, account_code) WHERE is_manual = false`, eliminating the silent-account-loss vector that caused the abandoned 44-07 deploy incident — without touching either of the two existing 3-arg callers.**

## Accomplishments

- Single migration file ships the entire structural fix in one atomic deploy unit:
  - **Backfill prologue** (D-44.1-05) — UPDATEs the 236 NULL `account_code` rows in production to `'ACCT-MISSING-' || id::text` so the partial unique index can be created without conflict.
  - **Partial unique index** `forecast_pl_lines_forecast_account_code_partial_uidx` on `(forecast_id, account_code) WHERE is_manual = false` — provides the conflict target for the UPSERT and structurally enforces uniqueness only over the derived (non-manual) row space.
  - **CREATE OR REPLACE FUNCTION** with the new 4-arg signature `(uuid, jsonb, jsonb, boolean DEFAULT false)` and the UPSERT body.

  - **DROP FUNCTION (Section 5b)** of the legacy 3-arg overload — removes PostgREST resolution ambiguity so the existing 3-arg callers in `forecast-wizard-v4/generate` and `forecast/[id]/recompute` resolve to the new 4-arg form via the DEFAULT.

## Why the previous DELETE-then-INSERT body was a data-loss vector

The body of `20260429000002_save_assumptions_and_materialize_rpc.sql` lines 73-117 executed:

```sql
DELETE FROM "public"."forecast_pl_lines"
 WHERE "forecast_id" = p_forecast_id
   AND "is_manual"   = false;

INSERT INTO "public"."forecast_pl_lines" (...)
SELECT ... FROM jsonb_array_elements(p_pl_lines) AS line;
```

When the upstream `convertAssumptionsToPLLines` produced an array shorter than DB state — empty assumption sub-section, undefined `year1Monthly`, shrunk `forecastDuration`, RLS-empty `existingLines` — the RPC silently lost accounts. There was no structural protection against shorter input. The wizard generate route + recompute route both pass exactly 3 args (no flag to opt in/out of full replace), so any shorter array was a silent destruction.

## The new UPSERT contract

```sql
INSERT INTO "public"."forecast_pl_lines" (...)
SELECT ... FROM jsonb_array_elements(p_pl_lines) AS line
ON CONFLICT ("forecast_id", "account_code") WHERE "is_manual" = false
DO UPDATE SET
  "account_name"     = EXCLUDED."account_name",
  "account_type"     = EXCLUDED."account_type",
  "account_class"    = EXCLUDED."account_class",
  "category"         = EXCLUDED."category",
  "subcategory"      = EXCLUDED."subcategory",
  "sort_order"       = EXCLUDED."sort_order",
  "actual_months"    = EXCLUDED."actual_months",
  "forecast_months"  = EXCLUDED."forecast_months",
  "is_from_xero"     = EXCLUDED."is_from_xero",
  "is_from_payroll"  = EXCLUDED."is_from_payroll",
  "computed_at"      = v_now,
  "updated_at"       = v_now;
```

- Accounts present in BOTH input and DB → updated in place (UPDATE branch of UPSERT).
- Accounts present in input but NOT in DB → inserted (INSERT branch).
- Accounts present in DB but NOT in input → **survive untouched** by the UPSERT, then receive a `computed_at = v_now` bump from Section 5 so the D-18 freshness invariant doesn't fire on the legitimate carry-forward.
- Manual rows (`is_manual = true`) → never touched by either path. The partial unique index excludes them, and every inserted row has `is_manual = false` hardcoded.

## The force_full_replace branch (D-44.1-02)

When callers genuinely need to clear all derived rows for a forecast (legitimate clear operations: year-type switch from FY → CY, intentional reset, etc.), they must opt in:

```ts
await supabase.rpc('save_assumptions_and_materialize', {
  p_forecast_id: ...,
  p_assumptions: ...,
  p_pl_lines: [],          // or the new full set
  p_force_full_replace: true
})
```

Then the body executes:

```sql
IF p_force_full_replace THEN
  DELETE FROM "public"."forecast_pl_lines"
   WHERE "forecast_id" = p_forecast_id
     AND "is_manual" = false;
END IF;
-- Then the same UPSERT runs (which will re-INSERT whatever the caller passed).
```

`is_manual = true` rows are still preserved in this branch (the DELETE includes `AND is_manual = false`).

**No current caller passes `p_force_full_replace=true`.** The existing 3-arg callers omit the parameter, get the DEFAULT `false`, and get the safe UPSERT semantics. The flag is reserved for future code that has a legitimate need (a year-type switcher in the wizard, for example).

## The legacy 3-arg overload was DROPPED

PostgreSQL treats different argument counts as different overloads. After `CREATE OR REPLACE FUNCTION` adds the 4-arg form, **both** the 3-arg and the 4-arg forms exist in the catalog. PostgREST resolves `supabase.rpc('save_assumptions_and_materialize', { p_forecast_id, p_assumptions, p_pl_lines })` by best-match — and the 3-arg overload (which still has the old DELETE-then-INSERT body) would be the preferred match for a 3-key body.

The migration includes:

```sql
DROP FUNCTION IF EXISTS "public"."save_assumptions_and_materialize"(uuid, jsonb, jsonb);
```

After the migration runs, the 4-arg form is the only resolution. The existing 3-arg callers keep working because the missing fourth argument defaults to `false`. **No code changes needed in `forecast-wizard-v4/generate/route.ts` or `forecast/[id]/recompute/route.ts`** (D-44.1-04 contract preserved).

## Decisions Made

See frontmatter `key-decisions` for the full list. Key highlights:

- **Backfill prologue is included** (not omitted) because the 44.1-01 audit confirmed N=236 null `account_code` rows in production. Without the prologue, the partial unique index creation would fail with a NULL-vs-NULL conflict on the first duplicate (or in this case, with the constraint check that PostgreSQL applies at index build time).
- **Tail computed_at bump (Section 5 of the body)** is the non-obvious correctness step: rows that exist in DB but are absent from `p_pl_lines` are preserved by the UPSERT (good), but they would keep their **old** `computed_at` (bad). When ForecastReadService asserts `computed_at >= updated_at`, those carry-forward rows would trip the freshness invariant. The tail UPDATE moves their `computed_at` forward to `v_now` so the assertion holds.
- **DROP of the legacy 3-arg overload is non-negotiable** for PostgREST resolution correctness. Without it, the 3-arg callers would silently keep using the OLD body.
