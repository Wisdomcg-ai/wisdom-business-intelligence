# Phase 44.1: atomic-save-hardening-and-staged-rollout — Context

<domain>
## Phase Boundary

Close out the abandoned 44-07 atomic save with a structural fix (not a count-threshold patch), then stage Wave 8/9 deploy behind a soft-fail invariant window so production read paths can't go red on legacy stale rows.

This phase exists because today's (Apr 28) deploy of Phase 44 Sub-phases A and B caused a misdiagnosed incident:
- **Misdiagnosis:** "atomic save removed data" (44-07's RPC).
- **Actual root cause:** Wave 5's `xero_pl_lines` wide → long migration broke 12 read paths (querying `monthly_values` column that no longer exists) AND dropped RLS policies on the renamed table. Both already hotfixed (`29730f6` wide-compat redirect, `ec5055e` RLS restore, `e641a8b` WIZARD_VERSION bump to clear stale localStorage).
- **Latent risk that remains:** 44-07's atomic-save RPC uses DELETE-all-then-INSERT-from-derived-array semantics. When the upstream `convertAssumptionsToPLLines` produces a shorter array than DB state (empty assumption sub-section, undefined `year1Monthly`, shrunk `forecastDuration`, RLS-empty existingLines), the RPC silently loses accounts. The current 6 tests in `save-and-materialize.test.ts` cover atomic-call shape, rollback, recompute auth/404 — none cover loss-by-shrinking-input.

After 44.1, D-12 atomic save semantics + D-18 freshness invariants are live in production with zero data-loss risk and zero false-positive 500s on legacy forecasts.

</domain>

<decisions>
## Implementation Decisions (locked in-session with user)

### RPC contract change (structural fix, not patch)
- **D-44.1-01:** Restructure `save_assumptions_and_materialize` from `DELETE WHERE is_manual=false` then INSERT → **UPSERT keyed on `(forecast_id, account_code)`**. Accounts not present in `p_pl_lines` survive untouched. The system structurally cannot lose accounts via shorter-input.
- **D-44.1-02:** Add an explicit `p_force_full_replace boolean DEFAULT false` parameter for legitimate clear operations (e.g., switching forecast_duration year-types, intentional reset). When `true`, falls back to old DELETE-then-INSERT semantics — caller must opt in.
- **D-44.1-03:** Manual-flag preservation logic stays: `is_manual = true` rows are preserved through both paths.
- **D-44.1-04:** RPC return shape is unchanged — `{forecast_id, computed_at, lines_count}` — to keep callers (44-07 generate route + recompute endpoint) untouched at the API surface.
- **D-44.1-05:** Keying on `account_code` requires that all rows have a non-null `account_code`. The current schema allows null — handle by: (a) requiring `account_code` to be non-null in the new RPC's INSERT/UPSERT (raise EXCEPTION on null), and (b) audit existing prod rows for null `account_code` and either backfill or document as known-limitation. Decision in research: which is right.

### Test coverage
- **D-44.1-06:** Add multi-vector regression tests in `src/__tests__/services/save-and-materialize.test.ts` covering each known loss vector:
  1. Empty assumption sub-section (`assumptions.opex.lines = []` while existingLines has 5 OpEx rows) → all 5 rows must survive.
  2. Undefined `year1Monthly` (no `forecast_months` data in input) → existing forecast_months must NOT be blanked.
  3. Shrunk `forecastDuration` (was 3, now 1) → Y2 + Y3 month keys must not be wiped from rows that already had them.
  4. RLS-empty `existingLines` (existingLines query returns `[]` due to RLS) → RPC must still preserve manual rows; non-manual rows can be replaced if input is truly the source of truth, but the test documents the behavior either way.
  5. Manual-flag duplicate insertion: a row exists with `(account_code='4000', is_manual=true)`, input passes `(account_code='4000')` (will become is_manual=false) → RPC must NOT create two rows for the same account_code.
- **D-44.1-07:** Existing 6 tests stay (atomic, rollback, non-fatal-swallowing, recompute success/auth/404). New tests are additive.

### Soft-fail mode for 44-08 invariants
- **D-44.1-08:** `ForecastReadService` D-18 invariants (`forecast_freshness`, `coverage_non_negative`) gate behind environment variable `FORECAST_INVARIANTS_STRICT` (default `false` during soak).
- **D-44.1-09:** When flag is `false`, invariant violation: log to Sentry with tag `invariant_violation_logged: true` AND breadcrumb data (forecast_id, computed_at, updated_at, delta_seconds), but RETURN the row anyway — no 500.
- **D-44.1-10:** When flag is `true`, current strict behavior — throws + Sentry tag `invariant: forecast_freshness` (or `coverage_non_negative`) + 500 to caller.
- **D-44.1-11:** Soak window: 24-48 hours of clean Sentry data (zero `invariant_violation_logged` events for active tenants) before flipping to `true`. Decision to flip is human, not automated.
- **D-44.1-12:** Wave 8 commit `76595c4` (the existing strict implementation) is amended OR a follow-up commit adds the gate — research determines cleanest path. Wave 9 commits (`d6c90bb`, `bc17883`, `e4f89d3`) don't need changes — they propagate whatever ForecastReadService throws.

## Specific Ideas

### UPSERT contract sketch
```sql
-- Pseudo-shape for research to validate
INSERT INTO forecast_pl_lines (...)
SELECT ... FROM jsonb_array_elements(p_pl_lines) AS line
ON CONFLICT (forecast_id, account_code) WHERE is_manual = false
DO UPDATE SET
  account_name = EXCLUDED.account_name,
  forecast_months = EXCLUDED.forecast_months,
  actual_months = EXCLUDED.actual_months,
  computed_at = v_now,
  updated_at = v_now,
  ...
```
Requires a partial unique index on `(forecast_id, account_code) WHERE is_manual = false`. Research: confirm this index doesn't already exist, draft the migration cleanly.

### Soft-fail flag pattern
```ts
// In ForecastReadService (sketch)
const STRICT = process.env.FORECAST_INVARIANTS_STRICT === 'true'

if (computedAt < updatedAt) {
  if (STRICT) {
    Sentry.captureException(new Error('INVARIANT VIOLATED ...'), { tags: { invariant: 'forecast_freshness' } })
    throw new InvariantError(...)
  } else {
    Sentry.addBreadcrumb({ category: 'invariant', data: { forecast_id, delta_seconds: ... } })
    Sentry.captureMessage('forecast_freshness violation (logging-only)', { tags: { invariant_violation_logged: 'forecast_freshness' } })
    // fall through — return the row
  }
}
```
