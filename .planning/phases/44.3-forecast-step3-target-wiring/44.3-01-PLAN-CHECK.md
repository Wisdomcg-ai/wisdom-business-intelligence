## §C. YTD month-key alignment (the hardest risk — planner-flagged)

**Claim under test:** RESEARCH.md §6 asserts that `currentYTD.revenue_lines[].by_month` keys align with `year1Monthly` keys because "YTD IS the current FY." The plan relies on this — if it's wrong for any non-July FY business, the YTD lock silently misses every completed month and Test 2 would only pass for July FYs.

**Direct source verification:**

1. **API side — `historical-pl-summary.ts:140-189`:**
   - Calls `calculateForecastPeriods(fiscalYear, yearStartMonth)` to get `actual_start_month` / `actual_end_month`.
   - Passes those to `aggregatePeriod` which builds `monthKeys` via raw calendar arithmetic from `actual_start_month` (line 222-228): `${current.getFullYear()}-${String(current.getMonth() + 1).padStart(2, '0')}`.
   - For Apr-Mar FY (yearStartMonth=4), FY2026, today=May 2026: `actual_start_month = '2025-04'`, monthKeys = `['2025-04', '2025-05', ..., '2026-04']`.
   - These calendar keys end up as `revenue_lines[].by_month` keys (line 294: `Object.fromEntries(monthKeys.map(mk => [mk, ...]))`).

2. **Wizard side — `useForecastWizard.ts:53-74` + `types.ts:568-577`:**
   - `generateMonthKeys(fiscalYearStart, yearStartMonth=7)` (legacy `types.ts` version, deprecated but still in use).
   - For wizard with `prev.fiscalYearStart = 2025` and **default `yearStartMonth = 7`** → produces `['2025-07', ..., '2026-06']`.
   - The wizard's `prev.fiscalYearStart` is the **calendar start year**, not the FY label.

3. **CRITICAL FINDING — non-July FY case:**
   - `generateMonthKeys` defaults `yearStartMonth = 7`. The wizard's `initializeFromXero` calls `generateMonthKeys(prev.fiscalYearStart)` at line 757 with NO yearStartMonth argument — meaning it always uses July FY.
   - For an Apr-Mar business (yearStartMonth=4), the wizard will generate `['2025-07', ..., '2026-06']` while the API returns `['2025-04', ..., '2026-03']` for the same FY.
   - **The keys would NOT align for non-July FY businesses.**

4. **BUT — this is a pre-existing misalignment, not introduced by 44.3:**
   - The current bug branch (lines 763-770) calls `remapMonthKeysToForecastYear(line.byMonth, prev.fiscalYearStart)` which **also** passes only `fiscalYearStart` to `generateMonthKeys` (via line 57). So the prior-year remap already uses July FY.
   - The empty-byLine fallback (lines 771-819) similarly defaults to July via `generateMonthKeys(prev.fiscalYearStart)` at line 757.
   - **Conclusion:** The wizard-v4 layer assumes July FY across the board today. Non-July FY businesses would already have broken Step 3 month-key alignment — that's a separate bug pre-dating 44.3, not something 44.3 introduces or worsens.
   - PHASE.md scope is "fix the target-aware distribution bug." Non-July FY support is a separate latent issue.

5. **For the active client base (per Phase 44 audit + memory.md):** Most coached clients use July FY (Australian businesses — JDS, Envisage, etc.). IICT-HK uses a different FY but is referenced in memory.md as a triage target, not a Step-3 user. The bug 44.3 fixes is reported by the operator and is the dominant case. Non-July FY misalignment would surface separately and as a different symptom (entire YTD missing, not "scaled to wrong target").

**Verdict for §C: PASS WITH NOTE.**

- The keys DO align for July FY businesses (the test scenario in Test 2).
- The keys do NOT align for non-July FY businesses, but this is a pre-existing latent issue in the wizard, NOT introduced by 44.3. The plan does not need an additional task to fix it — that would scope-creep into a new phase.
- **Recommended note for executor:** add a 1-line comment in the new code or test file noting "Wizard currently assumes yearStartMonth=7 (July FY) via `generateMonthKeys` default. Non-July FY support is out of scope for 44.3 — file a follow-up if a non-July client reports Step 3 issues." This prevents future engineers from assuming the fix is non-July-safe.
- **Recommended note for executor:** Test 2's PHASE.md success criterion #2 specifically references "JDS in May with YTD through April" — JDS uses July FY (verified in memory.md), so the test scenario is correct. The plan's Test 2 fixture (`fiscalYearStart = { year: 2025, month: 7 }`) matches.
