<objective>
Remove the lazy-create side effects from `BusinessProfileService.loadBusinessProfile` / `getOrCreateBusinessProfile`. These two methods are the owner_id read path used by `/business-profile` when no `activeBusiness` is set. Today they `.insert` a blank `businesses` row (name='My Business', owner_id=user.id) and a matching `business_profiles` row the first time any authenticated user visits `/business-profile` — including team members, coaches, and admins — which is the root cause of the Jessica @ Oh Nine phantom-business incident.

Purpose: Eliminate the side effect at the service layer so that page refactor in Plan 41-02 cannot regress, and so any OTHER caller of these methods (none currently, but future code paths) cannot accidentally create phantoms. Also strip the `'My Business'` string literal from ALL paths in this file (both owner path and coach path) so the whole-file grep-for-zero stays valid as a regression sentinel.

Output: `business-profile-service.ts` with the owner_id read path fully read-only, zero `'My Business'` literals anywhere in the file, and the coach path's insert preserved. `tsc --noEmit` exits 0.
</objective>
