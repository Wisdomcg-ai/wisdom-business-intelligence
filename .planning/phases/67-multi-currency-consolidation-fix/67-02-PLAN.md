# 67-02 — Route forecast wizard reads through the FX engine

## Goal

Step 2 prior-year, Step 3 current-FY actuals, and page-load YTD all show AUD-translated totals when the business has any non-AUD tenant. Single-tenant / all-AUD businesses remain **bit-identical**.

## Acceptance

- IICT forecast wizard Step 2 shows HKD lines as AUD; totals match `/api/monthly-report/consolidated`
- IICT forecast wizard Step 3 current-FY actuals also AUD-translated
- Single-tenant clients pass bit-identical regression lock (snapshot pre vs post)
- Feature-flagged via `FORECAST_FX_VIA_ENGINE` env var (`true` to enable; default off)

## Tasks

### Task 1 — Predicate helper

New: `src/lib/utils/needs-fx-consolidation.ts`

```ts
// Returns true iff any active included connection has functional_currency != 'AUD'.
// Cached per-request via the supplied supabase client; one DB read per business.
export async function needsFxConsolidation(supabase, businessId): Promise<boolean>
```

Uses `resolveBusinessIds` (existing) to handle the dual-ID system.

### Task 2 — Wire `historical-pl-summary.ts`

When `needsFxConsolidation(business_id)` is true:
- Call `runConsolidation(business_id, fiscalYear)` (the existing engine entry-point)
- Project the engine's per-line output to the existing `WideXeroRow[]` shape
- Aggregate via the existing `aggregatePeriod()` helper to produce `HistoricalPLSummary`

When false: existing direct-read path runs unchanged.

Wrap the new branch in `if (process.env.FORECAST_FX_VIA_ENGINE === 'true')` so it's a no-op until we flip the flag.

### Task 3 — Wire `forecast-read-service.ts`

`getMonthlyComposite()` is the higher-frequency read (page-load + Step 3). Apply the same predicate gate + engine projection. Returned `MonthlyComposite` shape must be unchanged — the wizard's downstream code is contract-stable.

### Task 4 — Audit saved `forecast_pl_lines` for IICT

`forecast_pl_lines` rows for IICT may already encode mixed-currency totals (saved before this fix). One-time diagnostic script: compare saved annual totals against engine-translated totals. If significant drift, write a migration script to re-materialize from current state.

### Task 5 — Regression lock test

Snapshot the wizard data for Envisage, JDS, Sydney Pressed Metal, Efficient Living (all single-tenant). With `FORECAST_FX_VIA_ENGINE=false` and `=true`, the wizard payloads MUST be byte-identical (engine should pass through for single-tenant).

### Task 6 — Multi-tenant FX integration test

Synthetic 3-tenant business (2 AUD + 1 HKD) with known fx_rates. Assert engine-translated totals match expected hand-calculation.

## Risk surfaces

| Risk | Mitigation |
|---|---|
| Engine has eliminations + account alignment passes not needed by single-tenant reads | Regression lock test catches any deviation |
| Engine's `presentation_currency: 'AUD'` hardcoded | Acceptable; parameterized in P5 |
| Saved `forecast_pl_lines` for IICT pre-fix are mixed-currency | Task 4 audit + one-time fix script |
| PR #209 (revenue Y1 cascade) + PR #211 (GP cascade) lock Y1 actual months from `currentYTD` — those values now FX-translated | Manual test on IICT; ensure cascade math still works |
| Performance — engine is heavier than direct read | Predicate gates single-tenant fast path; for multi-tenant, accept engine cost (it's the only correct answer) |

## Rollback

`FORECAST_FX_VIA_ENGINE=false` in Vercel env → reverts to existing direct-read path with no redeploy. Code paths coexist behind the flag.
