<objective>
Add a proactive, refresh-only Vercel cron that walks every active xero_connections row every 6 hours and calls getValidAccessToken on each one. The cron does NOT fetch Xero data — its single job is to keep the refresh-token alive (reset Xero's 60-day idle TTL 4× per day) and surface token-health problems via Sentry before users notice them.

Purpose:
- Refresh-only is faster, cheaper, and produces a CLEAN telemetry signal: a failure here means token problems, period (vs sync-all-xero where a failure could be Xero rate limits, transient API errors, or data-shape issues — all unrelated to token health).
- Resets the 60-day idle window 4× per day, so even completely dormant connections (paused clients, weekend stretches) stay alive indefinitely.
- Gives the rotation race fewer windows to land between long gaps — refresh happens proactively well before the 30-min access-token TTL forces an emergency refresh under user load.

Output: One new App Router route file, one vitest suite, one new entry in vercel.json crons array. No new dependencies, no schema changes, no UI.

CRITICAL ordering constraint (DO NOT MERGE this plan before 53-03 is live):
- This cron amplifies the refresh path across all connections every 6 hours. If shipped before 53-03's deactivation tightening, the rotation race + over-eager `unauthorized_client` deactivation that exists today would be triggered systematically across the entire portfolio — turning a sporadic JDS-style bug into a reliable phase-wide outage. 53-03 must be live in production before this plan deploys.
- Coordination with 53-02: 53-02 deletes src/app/api/Xero/refresh-tokens/route.ts. That deletion creates NO refresh gap — the deleted route had no scheduled invocation, and sync-all-xero's daily cron continues to refresh tokens as a side effect of its data fetch via getValidAccessToken (sync-orchestrator.ts:613). This plan's cron then takes over the dedicated refresh duty 4× per day.

Open question resolution (per phase research §"Open Questions" #1): Keep this cron SEPARATE from sync-all-xero. Reasoning embedded in the route docstring:
  (a) sync-all-xero does data fetch which can fail for unrelated reasons (Xero rate limit, transient API error, data-shape change) — confusing the signal "is this connection's token alive?";
  (b) refresh-only is much cheaper (~200ms per call) and can run more frequently without straining Xero or our function budget;
  (c) clearer telemetry — separate Sentry breadcrumbs/metrics let ops differentiate "token health" alerts from "sync pipeline" alerts.
</objective>

```typescript
export interface XeroConnection {
  id: string;
  business_id: string;
  tenant_id: string;
  tenant_name: string;
  access_token: string;
  refresh_token: string;
  expires_at: string;
  is_active: boolean;
  token_refreshing_at?: string | null;
}

export interface TokenRefreshResult {
  success: boolean;
  accessToken?: string;
  error?: TokenRefreshError;       // 'token_expired_permanently' | 'token_revoked' | 'network_error' | 'rate_limited' | 'server_error' | 'database_error' | 'unknown'
  message?: string;
  shouldDeactivate?: boolean;
}

export async function getValidAccessToken(
  connectionOrId: XeroConnection | { id: string },
  supabase: SupabaseClient
): Promise<TokenRefreshResult>;
```
Notes for status mapping:
- `success === true` AND token was already fresh (no Xero call) → `still_valid`. This is detectable because token-manager logs "Token still valid, expires at …" and short-circuits at line 99-105 BEFORE acquireRefreshLock. Since the result shape doesn't expose "did we actually call Xero?", the route should infer status by comparing the connection's `expires_at` BEFORE the call to `now + 15min` threshold. If pre-call expiry > now+15min, it was a `still_valid` short-circuit. Otherwise, a successful result is a `refreshed`.
- `success === true` AND pre-call expiry <= threshold → `refreshed`.
- `success === false` AND `shouldDeactivate === true` → `deactivated` (the token-manager already wrote is_active=false before returning; route just records the status).
- `success === false` AND `shouldDeactivate !== true` → `failed` (transient — retry will happen on next cron tick).
