<objective>
Replace the browser-side Supabase delete in the Integrations page with a hardened server-side disconnect endpoint that handles the **dual-ID system** correctly.

**The actual JDS failure mode** (per 53-RESEARCH §3): RLS does NOT block the FE delete — owner/coach `authenticated` sessions are permitted to DELETE on `xero_connections`. The bug is that `xero_connections` rows can exist under TWO different `business_id` values for the same business (one keyed by `businesses.id`, one keyed by `business_profiles.id`). The current FE does `.eq('business_id', businessId)` with a single ID — so one row gets deleted, the other survives, the FE flips to "disconnected" anyway, and the next sync attempt picks up the stale row.

This plan ships:

1. A new `POST /api/Xero/disconnect` route that authenticates the caller (mirroring the `/api/Xero/reactivate` cookie-session + RBAC pattern), resolves the input `business_id` to BOTH ID forms (canonical `businesses.id` + matching `business_profiles.id`), and uses the service-role Supabase client to issue a single `.delete().in('business_id', [bothIds])` against `xero_connections`. The response includes the actual deleted row count.

2. An updated `handleDisconnectXero` in `src/app/integrations/page.tsx` that calls the new route via `fetch()`. The FE only flips `setXeroConnected(false)` when the server returns `deleted_count > 0`; if 0 rows were deleted (or the request fails), the user sees an error and the connection indicator stays as it was.

3. A vitest suite that hits the route handler with a stub Supabase client and asserts the dual-ID delete is issued, plus auth and RBAC negative paths.

**Out of scope for this plan** (per PHASE.md sequencing):
- Token refresh changes (53-02 / 53-03)
- Proactive refresh cron (53-04)
- Sentry observability + dashboard health badge (53-05)

**Hard delete is used (not soft-delete).** 53-RESEARCH confirmed only `forecasts.xero_connection_id` references `xero_connections.id`, with `ON DELETE SET NULL` (`baseline_schema.sql:8785`). No CASCADE / RESTRICT pitfalls. Other Xero data tables (`xero_pl_lines`, `sync_jobs`, etc.) reference `business_id` not `xero_connections.id`, so deleting the connection row leaves their data untouched — which is correct: when the user reconnects, those historical rows remain useful as long as the new connection covers the same tenant.

Output: 1 new route file (~100 LOC), 1 new test file (~200 LOC), 1 modified component (~15 LOC change). 3 atomic commits. Single PR.
</objective>

### Auth + RBAC (verbatim from src/app/api/Xero/reactivate/route.ts:21-60)

```typescript
import { createRouteHandlerClient } from '@/lib/supabase/server';
import { createClient } from '@supabase/supabase-js';

const supabaseAdmin = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_KEY!
);

export const dynamic = 'force-dynamic';

export async function POST(request: NextRequest) {
  // 1. Cookie-session auth via the route-handler client
  const supabase = await createRouteHandlerClient();
  const { data: { user }, error: userError } = await supabase.auth.getUser();
  if (userError || !user) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }

  // 2. Body parse
  const { business_id } = await request.json();
  if (!business_id) {
    return NextResponse.json({ error: 'business_id is required' }, { status: 400 });
  }

  // 3. Lookup business — accept either businesses.id OR business_profiles.id as input
  //    (more permissive than reactivate; needed because the FE may pass either form
  //    depending on which loader produced businessId).
  let canonicalBusinessId: string | null = null;
  let profileBusinessId: string | null = null;

  // Try input as businesses.id
  const { data: bizDirect } = await supabaseAdmin
    .from('businesses')
    .select('id, owner_id, assigned_coach_id')
    .eq('id', business_id)
    .maybeSingle();

  let business: { id: string; owner_id: string | null; assigned_coach_id: string | null } | null = null;
  if (bizDirect) {
    business = bizDirect;
    canonicalBusinessId = bizDirect.id;
    const { data: profile } = await supabaseAdmin
      .from('business_profiles')
      .select('id')
      .eq('business_id', bizDirect.id)
      .maybeSingle();
    profileBusinessId = profile?.id ?? null;
  } else {
    // Try input as business_profiles.id
    const { data: profile } = await supabaseAdmin
      .from('business_profiles')
      .select('id, business_id')
      .eq('id', business_id)
      .maybeSingle();
    if (profile?.business_id) {
      profileBusinessId = profile.id;
      canonicalBusinessId = profile.business_id;
      const { data: bizFromProfile } = await supabaseAdmin
        .from('businesses')
        .select('id, owner_id, assigned_coach_id')
        .eq('id', profile.business_id)
        .maybeSingle();
      if (bizFromProfile) business = bizFromProfile;
    }
  }

  if (!business) {
    return NextResponse.json({ error: 'Business not found' }, { status: 404 });
  }

  // 4. RBAC: owner OR assigned coach OR super_admin (mirror reactivate exactly)
  let allowed = business.owner_id === user.id || business.assigned_coach_id === user.id;
  if (!allowed) {
    const { data: roleRow } = await supabase
      .from('system_roles')
      .select('role')
      .eq('user_id', user.id)
      .maybeSingle();
    if (roleRow?.role === 'super_admin') allowed = true;
  }
  if (!allowed) {
    return NextResponse.json({ error: 'Access denied' }, { status: 403 });
  }

  // 5. Build the dual-ID delete target list (drop nulls, drop dupes)
  const idsToDelete = Array.from(
    new Set([canonicalBusinessId, profileBusinessId].filter(Boolean) as string[])
  );

  // 6. Hard DELETE via service role (bypasses RLS — explicit by design).
  //    Use .select() to get back the deleted rows so we can return a count.
  const { data: deletedRows, error: deleteError } = await supabaseAdmin
    .from('xero_connections')
    .delete()
    .in('business_id', idsToDelete)
    .select('id, business_id, tenant_id, is_active');

  if (deleteError) {
    console.error('[Xero Disconnect] Delete failed:', deleteError);
    return NextResponse.json({
      success: false,
      error: 'delete_failed',
      message: deleteError.message,
    }, { status: 500 });
  }

  const deleted_count = deletedRows?.length ?? 0;

  if (deleted_count === 0) {
    // Not a hard failure (the row may have been pre-cleaned), but the FE should
    // NOT optimistically flip to 'disconnected' — surface a clear message.
    return NextResponse.json({
      success: false,
      error: 'nothing_to_delete',
      message: 'No Xero connections found for this business under either ID form. The connection may have already been removed.',
      deleted_count: 0,
      ids_checked: idsToDelete,
    }, { status: 200 });
  }

  console.log('[Xero Disconnect] Deleted', deleted_count, 'rows for business', business_id, 'across IDs', idsToDelete);

  return NextResponse.json({
    success: true,
    deleted_count,
    deleted_ids: idsToDelete,
    deleted_rows: deletedRows.map(r => ({
      id: r.id,
      business_id: r.business_id,
      tenant_id: r.tenant_id,
      was_active: r.is_active,
    })),
  });
}
```

## Updated FE handler in src/app/integrations/page.tsx (replace lines 95-115)

```typescript
async function handleDisconnectXero() {
  if (!businessId || !confirm('Are you sure you want to disconnect Xero?')) return;

  setSyncing(true);
  try {
    const res = await fetch('/api/Xero/disconnect', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ business_id: businessId }),
    });

    const data = await res.json();

    // Treat success only when the server confirms at least one row was deleted.
    // Phase 53-01: do NOT optimistically flip — JDS taught us that flipping
    // before the server confirms hides dual-ID stale rows.
    if (res.ok && data.success && (data.deleted_count ?? 0) > 0) {
      setXeroConnected(false);
      setXeroData(null);
    } else {
      const message = data.message || data.error || 'Failed to disconnect Xero';
      console.error('[Integrations] Disconnect failed:', { status: res.status, data });
      alert(message);
    }
  } catch (error) {
    console.error('Error disconnecting Xero:', error);
    alert(error instanceof Error ? error.message : 'Failed to disconnect Xero');
  } finally {
    setSyncing(false);
  }
}
```
