<objective>
Build the FOUNDATION the rest of Phase 74 depends on, with ZERO behaviour change to existing
code: (1) a single `resolveBusinessProfileId()` that turns any of the three id-spaces into the
canonical `business_profiles.id` (or null), and (2) a tiny `surfaceSupabaseError()` helper so the
swallowed FK/PostgREST errors (which hid all 23 bugs) get logged instead of discarded. The branded
`BusinessProfileId` type already exists in `src/lib/types/ids.ts` — this plan USES it; it does not
re-add it.
</objective>

<context>
Canonical key for the affected tables = `business_profiles.id`. `businesses ↔ business_profiles`
is 1:1 (verified), and the two UUID namespaces are disjoint (0 collisions) — so an id can be
classified by probing the tables. `business_profiles` links to `businesses` via its `business_id`
column (FK) and to the owner via `user_id`. Existing `resolveBusinessId()` returns a
`businesses.id` and is correct for businesses-keyed tables — do NOT change it.
</context>

<action>
Create `src/lib/business/resolveBusinessProfileId.ts` exporting:

  export async function resolveBusinessProfileId(
    supabase: SupabaseClient,
    input: string | null | undefined,
  ): Promise<BusinessProfileId | null>

Logic (return on first hit; null on miss — NEVER return a businesses.id):
  1. if !input → return null
  2. already a profile id?  `business_profiles.select('id').eq('id', input).maybeSingle()` → if row, return toBusinessProfileId(row.id)
  3. a businesses.id?       `business_profiles.select('id').eq('business_id', input).maybeSingle()` → if row, return toBusinessProfileId(row.id)
  4. a user_id?            `business_profiles.select('id').eq('user_id', input).maybeSingle()` → if row, return toBusinessProfileId(row.id)
  5. return null
On any supabase error, call surfaceSupabaseError('resolveBusinessProfileId', error) (task 2) and continue to the next probe / return null.

Write `src/__tests__/business/resolveBusinessProfileId.test.ts` (vitest, mock supabase like the existing annual-reset tests): cases — input is a profile id (returns it); input is a businesses.id (returns the linked profile id); input is a user_id (returns the owner's profile id); unknown id (returns null); null/undefined input (returns null); never returns the businesses.id even when input is one.
</action>
