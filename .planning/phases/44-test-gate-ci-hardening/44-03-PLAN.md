Rewrite `.github/workflows/supabase-preview.yml` to the EXACT content below.
Use the Write tool. Keep the workflow `name: Supabase Preview Branch` and the
same `on:` trigger so existing branch-protection rules continue to recognise
it as the "Supabase Preview Branch / &lt;job&gt;" check name.

Each job:
- Runs on `ubuntu-latest`
- Uses `actions/checkout@v4` then `actions/setup-node@v4` with Node 20 + npm cache
- Runs `npm ci` (not `npm install`) for reproducible installs
- Has `timeout-minutes` set conservatively per job
- Has a clear `name:` so PR status checks are readable

```yaml
name: Supabase Preview Branch

# Sanity-check PRs before Supabase auto-creates a preview branch.
# Each job below is a separate required status check on `main`:
#   - migration-check: migration filenames are well-formed
#   - lint:            next lint passes (TEST-02)
#   - typecheck:       tsc --noEmit passes (TEST-03)
#   - vitest:          vitest run passes (TEST-04)
#   - build:           next build succeeds (TEST-05)
# Jobs run in parallel; each fails independently so PR authors see exactly
# which gate broke.
#
# The actual preview-branch DB is spun up by Supabase's GitHub integration,
# not this workflow.

on:
  pull_request:
    branches: [main]
    paths:
      - 'supabase/**'
      - 'src/**'
      - 'package.json'
      - 'package-lock.json'
      - 'next.config.js'
      - 'tsconfig.json'
      - '.eslintrc.json'
      - 'vitest.config.ts'
      - '.github/workflows/supabase-preview.yml'

jobs:
  migration-check:
    name: migration filenames
    runs-on: ubuntu-latest
    timeout-minutes: 2
    steps:
      - uses: actions/checkout@v4
      - name: Check migration filenames
        run: |
          INVALID=$(find supabase/migrations -maxdepth 1 -name '*.sql' -type f \
            | grep -v -E '^supabase/migrations/[0-9]{14}_[a-z0-9_]+\.sql$' \
            | grep -v -E '^supabase/migrations/[0-9]{8}_[a-z0-9_]+\.sql$' || true)
          if [ -n "$INVALID" ]; then
            echo "Invalid migration filenames (must be YYYYMMDD_name.sql or YYYYMMDDHHMMSS_name.sql):"
            echo "$INVALID"
            exit 1
          fi
          echo "All migration filenames are valid."

  lint:
    name: lint
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: ESLint (next lint)
        run: npm run lint

  typecheck:
    name: typecheck
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: TypeScript (tsc --noEmit)
        run: npx tsc --noEmit

  vitest:
    name: vitest
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: Vitest
        run: npx vitest run --reporter=dot

  build:
    name: build
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: Next build
        run: npm run build
        env:
          # next build needs SOME value for the public Sentry DSN to bundle without
          # warnings; provide a placeholder. The Sentry source-map upload is
          # disabled in next.config.js when SENTRY_AUTH_TOKEN is unset.
          NEXT_PUBLIC_SENTRY_DSN: 'https://example@sentry.io/0'
          # Skip Sentry CLI work in CI builds (no auth token available)
          SENTRY_AUTH_TOKEN: ''
```

Notes on choices:
- `npm ci` (not `npm install`) so failures from a stale lockfile are caught early.
- Each job re-runs `npm ci` rather than sharing a build artifact between jobs.
  Simpler, ~30 seconds slower per job, no actions/cache complexity. Optimise
  later if total wall time becomes painful.
- `paths:` was extended to include `next.config.js`, `tsconfig.json`,
  `.eslintrc.json`, `vitest.config.ts`, and the workflow itself — changes to
  any of these should re-trigger validation.
- `name:` on each job uses lowercase single words so the PR status check labels
  are clean: `Supabase Preview Branch / lint`, `Supabase Preview Branch / build`, etc.
- The `build` job's `env:` block provides placeholder values for build-time
  env vars that would otherwise warn or fail. Real production secrets are NOT
  needed for `next build`; the build only needs to succeed compilation, not
  actually contact Sentry. Document this in the SUMMARY.
- The migration-filename job intentionally does NOT install Node — it's a
  pure shell+grep check, finishes in seconds.

DO NOT:

- Add a Playwright job here (Playwright runs in a separate nightly workflow — see Plan 44-04)
- Add `npm audit` here (deferred per audit Strategic Investments #10 to a future phase)
- Add coverage thresholds (PHASE.md "out of scope")
- Combine jobs back into a single `validate` job (defeats TEST-02..05 separation)
