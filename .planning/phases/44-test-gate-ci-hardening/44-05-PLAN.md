<objective>
Final phase step: configure branch protection on `main` to mark the four PR
gates as Required status checks, set the `PLAYWRIGHT_BASE_URL` secret so the
nightly workflow has a target, then verify the gates actually block bad PRs.

Purpose: Without branch protection, the four green checks from Plan 44-03 are
informational only — anyone can merge a red PR. This plan flips them to
required and proves the gate works by deliberately breaking it.

Output: Branch-protection rule updated, secret provisioned, three throwaway
PRs (lint-fail, tsc-fail, build-fail) demonstrating each gate blocks merging.
The phase deliverable is verified end-to-end.

This plan is NOT autonomous — it requires human action in the GitHub UI for
both branch protection and secret provisioning. The `gh` CLI requires admin
permissions on the repo and we do not script that path here per the phase
scope.
</objective>

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 2: Configure branch protection on `main` — mark 4 checks Required</name>
  <read_first>
    - .github/workflows/supabase-preview.yml (the workflow file — confirm the four job names: `lint`, `typecheck`, `vitest`, `build`)
    - .planning/phases/44-test-gate-ci-hardening/44-03-SUMMARY.md (which should record the wall-clock job times so the user knows what to expect on PR runs)
  </read_first>
  <what-built>
    Plan 44-03 split the validation workflow into five separate jobs. Four of
    them (lint, typecheck, vitest, build) are the merge gates. They currently
    run on every PR but are NOT marked as required — a developer with merge
    permissions can override a red gate. This task makes them required.
  </what-built>
  <how-to-verify>
    1. Open https://github.com/Wisdomcg-ai/wisdom-business-intelligence/settings/branches
    2. If a rule already exists for `main`: click "Edit". If not: click "Add classic branch protection rule" (or the equivalent if the repo is on the new ruleset model — see Note A below).
    3. Branch name pattern: `main`
    4. Under "Protect matching branches", check:
       - [x] Require a pull request before merging
       - [x] Require status checks to pass before merging
       - [x] Require branches to be up to date before merging (recommended; ensures stale PRs re-run CI)
    5. In the "Status checks that are required" search box, type and add EACH of:
       - `lint`
       - `typecheck`
       - `vitest`
       - `build`
       (These are the `name:` values from `supabase-preview.yml`. If GitHub shows them prefixed with `Supabase Preview Branch /`, that's expected — pick those entries.)
    6. (Optional but recommended) Also check:
       - [x] Do not allow bypassing the above settings (so even repo admins can't merge a red PR)
    7. Click "Create" / "Save changes"
    8. Verify by opening any open PR (or creating a draft one): the merge button should grey out until those four checks pass.

    **Note A — classic vs ruleset model:** If the repo settings show "Rules" /
    "Rulesets" instead of "Branch protection rules", use the same logic in the
    Rulesets UI: target = `main`, enforcement = active, rules = "Require status
    checks to pass" with the same four checks. The functional outcome is identical.

    **Note B — DO NOT add `migration filenames` as a required check.** It only
    runs when `supabase/**` paths change, so requiring it would block PRs that
    don't touch migrations.

    **Note C — DO NOT mark the Playwright nightly run as required.** It's a
    nightly cron job, not a per-PR check, so requiring it would prevent every
    PR from merging.
  </how-to-verify>
  <acceptance_criteria>
    - The branch protection rule on `main` lists exactly these four required checks: `lint`, `typecheck`, `vitest`, `build`
    - The migration-filename check is NOT marked as required (because of path-filter behaviour)
    - The Playwright nightly run is NOT marked as required (it's a nightly cron, not a PR gate)
    - Opening a draft PR shows all four checks listed in the status panel and merge is gated on them passing
  </acceptance_criteria>
  <resume-signal>
    Reply "branch protection updated" with a screenshot or copy-paste of the
    rule configuration (the list of required checks).
  </resume-signal>
</task>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 3: Prove each gate blocks a bad PR (smoke test the protection)</name>
  <read_first>
    - .github/workflows/supabase-preview.yml (so the user knows what each check is testing)
  </read_first>
  <what-built>
    Branch protection is now configured (Task 2). This task proves each of the
    four gates actually blocks a merge by deliberately breaking each one in a
    throwaway PR.
  </what-built>
  <how-to-verify>
    Open ONE branch from `main` named `phase-44/protection-smoke`. Make THREE
    sequential commits, each one breaking exactly ONE gate. After each commit:
    push, observe the PR status, then revert the change before making the next.
    Document the result of each.

    **Smoke 1 — lint break:**
    1. In a frequently-imported file (pick something like `src/app/page.tsx`),
       add a deliberate lint violation. The simplest:
       ```ts
       // Add at the top of the file
       const _unused: any = {}   // eslint-disable-line — JK no, no disable; let it fire
       ```
       Or simpler: introduce a `let x = 1; x = 2;` that triggers `prefer-const`
       (which is on by default in `next/core-web-vitals`).
    2. Commit, push, open PR.
    3. Confirm the `Supabase Preview Branch / lint` check goes red and the
       merge button is greyed out.
    4. Revert the change locally, commit, push.
    5. Confirm `lint` goes green and merge is unblocked.

    **Smoke 2 — typecheck break:**
    1. In the same file, introduce a TS error: `const x: number = "string"`.
    2. Commit, push.
    3. Confirm `Supabase Preview Branch / typecheck` goes red. Merge is blocked.
    4. Revert.

    **Smoke 3 — build break:**
    1. In the same file, import a non-existent module: `import './does-not-exist'`.
    2. Commit, push.
    3. Confirm `Supabase Preview Branch / build` goes red. Merge is blocked.
       (`typecheck` may also go red — that's fine, both should reject this.)
    4. Revert.

    **Smoke 4 — vitest break (optional but recommended):**
    1. Add a deliberately failing test in `src/__tests__/protection-smoke.test.ts`:
       ```ts
       import { test, expect } from 'vitest'
       test('this should fail', () =&gt; {
         expect(1).toBe(2)
       })
       ```
    2. Commit, push.
    3. Confirm `Supabase Preview Branch / vitest` goes red. Merge is blocked.
    4. Delete the file, commit, push.

    **Cleanup:** Close the PR without merging. Delete the
    `phase-44/protection-smoke` branch. The smoke is a verification ritual —
    nothing from this branch should land on `main`.
  </how-to-verify>
  <acceptance_criteria>
    - For ALL FOUR gates (lint, typecheck, vitest, build): a deliberate break caused the corresponding PR check to go red and the merge button to be unavailable. Vitest is broken by adding a one-line failing test (e.g., `it('intentional vitest smoke fail', () => expect(true).toBe(false))`) and reverting it after the gate goes red.
    - The smoke PR is closed without merging
    - The smoke branch is deleted (or noted as deletable in the SUMMARY)
    - For each gate: SUMMARY records the commit SHA that broke it and the URL of the failed CI run, as evidence
    - The `playwright-nightly.yml` workflow has produced at least one green run (per Task 1) — OR a failed first run has been triaged
  </acceptance_criteria>
  <resume-signal>
    Reply "gates verified" with the list of failed-CI run URLs (one per smoked gate).
    OR "gates verified, vitest skipped" if you chose to skip Smoke 4.
    OR a description of any gate that did NOT block as expected, so we can fix it.
  </resume-signal>
</task>
