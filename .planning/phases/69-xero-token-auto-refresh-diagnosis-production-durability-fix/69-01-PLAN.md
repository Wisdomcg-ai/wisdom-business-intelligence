Perform a structured read-only investigation. For EACH of the 7 hypotheses listed in 69-CONTEXT.md `<decisions>` section, gather evidence and emit a verdict (CONFIRMED / RULED OUT / INCONCLUSIVE) with a one-line justification linked to evidence.

**The 7 hypotheses (verbatim from 69-CONTEXT.md):**
1. Cron not firing: Vercel cron schedule misconfigured or disabled
2. Cron firing but erroring silently: refresh endpoint throws, no Sentry capture
3. Refresh succeeding but not persisting: token written to memory but not back to `xero_connections` (RLS, transaction failure, supabase-js auth-lock interaction)
4. Refresh threshold too late: cron only refreshes within 15min of expiry (per `REFRESH_THRESHOLD_MINUTES = 15` in token-manager.ts:23); if cron itself fails once, token already dead by next run
5. Per-tenant fail-forward gap: one tenant's refresh failure aborts the whole batch instead of isolating + continuing
6. Token rotation issue: Xero returns new `refresh_token` on rotation; if we don't persist it, next refresh fails
7. fec0c1e2 auth-lock cap regression: 10s cap aborts a slow refresh write mid-transaction

**Investigation protocol — for each hypothesis, run the listed checks and record evidence verbatim in the diagnosis doc:**

### H1 — Cron firing?
- Read vercel.json — confirm `/api/cron/refresh-xero-tokens` exists at `0 */6 * * *` UTC. Record the literal entry.
- Run: `vercel logs --since 7d https://wisdombi.ai 2>&1 | grep -i "refresh-xero-tokens" | head -50` OR `vercel ls && vercel logs <production-deployment-url> --since 7d 2>&1 | grep refresh-xero-tokens`. Record the last 5 invocation timestamps + HTTP status. If vercel CLI is unavailable in this environment, document the exact command Matt needs to run and capture the output by asking.
- Verdict: CONFIRMED FIRING if ≥4 invocations/day for last 7 days. RULED OUT if zero invocations. INCONCLUSIVE if Matt cannot pull logs.

### H2 — Cron erroring silently?
- Run Sentry MCP query: events tagged `invariant:cron_refresh_xero_tokens` in last 14 days. Record event count + last 3 stack traces verbatim.
- Run Sentry MCP query: events tagged `invariant:cron_refresh_xero_tokens_per_connection` in last 14 days. Record count.
- Run Sentry MCP query: events tagged `invariant:cron_refresh_xero_tokens_failed` in last 14 days. Record count + last 3 messages.
- Verdict: CONFIRMED if any aggregate error events. RULED OUT if zero events AND H1 confirms cron is firing.

### H3 — Refresh succeeding but not persisting?
- Read `src/lib/xero/token-manager.ts:401-422` (the `.update().eq('id', connectionId)` block). Confirm there is NO `.select()` or returned-row check after the update — i.e. update errors are caught (line 412) but a zero-row update would not throw. Document this.
- Run SQL query against production (Matt's existing tooling, e.g. supabase MCP or scripts/phase-70-data-audit.mjs equivalent):
  ```sql
  SELECT id, business_id, tenant_name, expires_at, updated_at,
         token_refreshing_at, is_active
  FROM xero_connections
  WHERE business_id IN (
    '8c8c63b2...', -- Envisage
    'fea253dd...', -- JDS
    'fbc6dffd...'  -- IICT (3 tenants)
  )
  ORDER BY business_id, tenant_name;
  ```
  (Substitute the real UUIDs from 69-CONTEXT.md table. If only the prefix is available, run a `LIKE '8c8c63b2%'` form. Use the exact tenant breakdown from the audit table.)
  Record for each row: `expires_at`, `updated_at`, `token_refreshing_at`, `is_active`.
- Compare `updated_at` against the most recent expected cron tick (e.g. 2026-05-30 00:00 UTC). If `updated_at` is days behind `now()` but `is_active=true`, the cron is either not touching the row OR the update is silently no-op'ing.
- Check for stuck `token_refreshing_at` (lock held > 30s from now). If non-null and old, the cron acquired the lock but never released it.
- Verdict: CONFIRMED if `updated_at` lags cron schedule by >12h on multiple rows. RULED OUT if `updated_at` advances on every cron tick.

### H4 — Refresh threshold too late?
- Note: `REFRESH_THRESHOLD_MINUTES = 15` and Xero access tokens last 30min. Cron at `0 */6 * * *` = every 360min. Refresh-token has 60-day idle TTL.
- Walk the timeline: at T=0 cron refreshes (new access token valid 30min, expires at T+30). At T+360 (next cron), access token is already long-expired but the cron should still call getValidAccessToken which goes through refresh path. Confirm this matches code at refresh-xero-tokens/route.ts:160-170 + token-manager.ts:205-213 (the `if (expiry > thresholdTime)` short-circuit). Document the expected behaviour: cron should refresh on EVERY tick because access token TTL (30min) << cron interval (360min).
- Verdict: usually RULED OUT — threshold is irrelevant if cron is firing every 6h and access token is 30min. But if H1 confirms cron NOT firing, this becomes structurally moot.

### H5 — Per-tenant fail-forward gap?
- Read refresh-xero-tokens/route.ts:152-247. Confirm per-row try/catch wraps the entire iteration body. Document the try/catch boundary line numbers.
- Read token-manager.ts — note that `getValidAccessToken` returns `{success, error, shouldDeactivate}` shapes; it does NOT throw on Xero errors (it categorizes them). Document this.
- Verdict: RULED OUT by code reading IF the try/catch is intact and broad. If any path can throw outside the catch (e.g. supabase client creation), document the gap.

### H6 — Token rotation issue?
- Read token-manager.ts:394-410 (the success-path update). Confirm `refresh_token: encrypt(tokens.refresh_token)` is in the update payload. Document the literal line.
- Read callback/route.ts:60-79 (OAuth save). Confirm refresh_token is persisted on initial connect. Document literally.
- Verdict: RULED OUT if both write the new refresh_token. CONFIRMED if either path drops it.

### H7 — fec0c1e2 auth-lock cap regression?
- Read `git show fec0c1e2 --stat` — list every file touched. Confirm only browser-side files: `src/lib/supabase/client.ts` + login pages + `src/lib/auth/lock-recovery.ts`. Document the file list literally.
- Read `src/lib/supabase/admin.ts` — confirm the service role client the cron uses does NOT call `createBrowserClient` and does NOT inherit `lockAcquireTimeout`. Document the createClient call literally.
- Verdict: RULED OUT if cron uses admin.ts (server-side) and admin.ts is untouched. CONFIRMED only if cron somehow reaches the browser client (would be a routing bug).

### Bonus check — RLS interaction
The cron uses the service role client (`createServiceRoleClient`) which bypasses RLS by design. Confirm this in `src/lib/supabase/admin.ts` — the secret key bypasses RLS. Document. If for any reason the cron is using the anon client, the update would silently no-op under RLS.

After the 7 hypotheses + bonus, name 1-3 ROOT CAUSES with evidence references. If evidence is insufficient to name a root cause, name the most likely 1-2 and recommend a 1-day instrumented production capture to confirm.
