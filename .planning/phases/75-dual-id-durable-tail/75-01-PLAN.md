<objective>
Prove the data is FK-ready BEFORE any FK is added in 75-02. An FK to business_profiles(id) REJECTS
rows keyed by businesses.id or user_id, so this wave (1) audits every profile-keyed FK-target column
for wrong-id-space rows and text→uuid cast safety, and (2) backfills the few known gaps (guarded,
snapshot-first). Output is a go/no-go report: "0 mis-keyed rows + every FK-target column clean".
Read-mostly; the only writes are narrow, snapshotted backfills with operator sign-off.
</objective>

<context>
Canonical key = business_profiles.id (1:1 with businesses; disjoint UUID namespaces).
FK-target columns (from FK-INTEGRITY-PLAN.md, Phase B scope):
  - text business_id, need uuid cast: activity_log (2545), plan_snapshots (58), sprint_key_actions (3),
    kpi_history (0).
  - dual-column (text business_id + uuid business_profile_id): business_financial_goals (14),
    business_kpis (55) — FK will go on business_profile_id, so it MUST be fully populated.
  - Group B → businesses(id) (uuid, no orphans expected): issues_list, open_loops, strategy_data,
    cashflow_assumptions.
Existing tooling: scripts/audit-dual-id-full.mjs + scripts/audit-dual-id-distribution.mjs (untracked,
read-only) already exist — EXTEND them, do not duplicate. Per DUAL-ID-VERIFIED.md prod was already
~clean (55/55 business_kpis profile-keyed); this wave confirms that empirically and closes any gap.
</context>

<task number="1" name="FK-readiness audit (read-only)">
<read_first>
- scripts/audit-dual-id-full.mjs + scripts/audit-dual-id-distribution.mjs (existing patterns + prod client setup)
- .planning/codebase/FK-INTEGRITY-PLAN.md (the exact table/column list + counts)
- src/lib/business/resolveBusinessProfileIds.ts (canonical resolver — reuse for classification)
</read_first>
<action>
Write scripts/audit-dual-id-fk-readiness.mjs (read-only; no flags = report). For EACH FK-target column:
  - Profile-keyed (Group A): count rows whose business_id is NOT present in business_profiles.id
    (i.e. keyed by businesses.id or user_id, or orphaned). Report per-table counts + sample ids.
  - Text columns: assert every business_id is a syntactically valid uuid AND exists in
    business_profiles.id (cast-safety + FK-safety in one).
  - Dual-column tables: count rows where business_profile_id IS NULL (the FK-blockers) and whether the
    legacy text business_id resolves to a profile id for each.
  - Group B (businesses-keyed): count rows whose business_id is not in businesses.id (orphans).
Emit a machine-readable summary line per table: `TABLE | total | wrong_key | null_profile_id | uncastable`.
</action>
<acceptance_criteria>
- `node scripts/audit-dual-id-fk-readiness.mjs` runs read-only against prod and exits 0
- Output covers all 10 FK-target tables with the 4 counts each
- Zero writes (grep the script: no insert/update/delete/upsert)
</acceptance_criteria>
</task>
