# Phase 28.0 — Quick Wins + Test Suite

## Objective

Fix three active engine bugs that are materially affecting cashflow accuracy for
current clients, and lock in current correct behaviour with a test suite before
any subsequent structural refactoring happens in 28.1–28.4.

## Scope

### Fix #1 — OpEx paid in accrual month (Calxa Rule 7)

**Current behaviour** (`engine.ts:418`):
```ts
// ALL OpEx except employment/bank fees gets DPO spread
for (const split of dpoSplit) {
  const targetMonth = allMonths[i + split.offset]
  if (targetMonth) cashOpExPayments[targetMonth] += cashAmount * split.portion
}
```

**New behaviour**:
```ts
// OpEx paid in the month it's accrued (rent, utilities, marketing, etc.)
// Employment and bank fees already paid immediately — this extends to all OpEx
cashOpExPayments[monthKey] += cashAmount
```

**Why**: Calxa Rule 7 — "Debtor distribution applies to revenue, creditor
distribution applies to COGS only. Opex is paid immediately unless account has a
profile override." Per-account overrides come in Phase 28.3.

**Impact**: Cashflow will show higher outflows in accrual months and lower
outflows in subsequent months. Net effect is more accurate monthly bank movement.

### Fix #2 — DSO/DPO math overlap in `getTimingSplit`

**Current behaviour** (pseudocode):
```ts
// DSO=45 can produce: [{offset:1, portion:1.0}, {offset:2, portion:0.5}]
// Total = 150% — allocates MORE than 100% of revenue
```

**New behaviour**: Strict bucket-based split matching Calxa's approach:
```ts
function getTimingSplit(days: number): {offset: number, portion: number}[] {
  const bucket = Math.floor(days / 30)
  const fraction = (days % 30) / 30
  const result = []
  if (1 - fraction > 0) result.push({offset: bucket, portion: 1 - fraction})
  if (fraction > 0) result.push({offset: bucket + 1, portion: fraction})
  // Guaranteed sum = 100%
  return result
}
```

**Impact**: DSO/DPO splits now sum to exactly 100%, preventing over-allocation
in high-day scenarios.

### Fix #3 — Depreciation add-back (keyword-based, interim)

**Current behaviour**: Depreciation accounts (typically in OpEx section) flow
through as cash outflows. Wrong — depreciation is non-cash.

**New behaviour**: Identify depreciation accounts by keyword (`depreciation`,
`amortisation`, `amortization`) and exclude them from cash outflows. Track as
a separate "Non-cash adjustments" line in output for transparency.

**Why interim**: Keyword matching is fragile. Phase 28.2 upgrades to explicit
account ID once the settings foundation (28.1) exists.

**Impact**: Bank balance no longer artificially depressed by depreciation.
