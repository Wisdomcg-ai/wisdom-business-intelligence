<objective>
**The highest-risk batch in Phase 49.** CASCADE is the only irreversible operation — once a delete fires in production, the cascaded children CANNOT be recovered. Per RESEARCH.md cross-cutting rollback section (lines 700-707): "DB-04 CASCADE is **irreversible if a delete fired in production.** This is why DB-04 CASCADE PRs (49-06) MUST be tested on a preview branch first."

Apply ON DELETE CASCADE to the ~5 Bucket B FKs (per `docs/db/fk-policy.md`):
- `process_flows.from_step_id → process_steps.id` (baseline:9130) — flow without source step is nonsensical
- `process_flows.to_step_id → process_steps.id` (baseline:9140) — flow without target step is nonsensical
- `process_flows.process_id → process_diagrams.id` (baseline:9135) — flow belongs to its diagram
- `process_phases.process_id → process_diagrams.id` (baseline:9145) — phase belongs to its diagram
- `session_attendees.user_id → auth.users.id` (baseline:9285) — attendance record without a user is meaningless (Bucket C-3 in RESEARCH.md but rebucketed to B per RESEARCH.md line 309 recommendation)

**Why CASCADE not SET NULL for these specific FKs:** the child rows have NO meaning without their parent. A `process_flows` row that points at a deleted step is a referential error — there's no UX or business rationale for an orphan flow. Similarly an `annual_snapshots` row CAN survive without a quarterly child (those went to Bucket A SET NULL), but a `process_flows` row CANNOT. The migration COMMENT for each CASCADE FK must spell out this rationale so future reviewers can understand the choice.

**Why preview-branch testing is extra-thorough here:** for each CASCADE FK, two assertions:
1. **Immediate cascade fires** — child row is gone after parent delete.
2. **Cascade chain is bounded** — no surprise grandchild deletions, no unrelated rows affected, no orphans surviving elsewhere.

The 2nd assertion is the new one. RESEARCH.md notes the existing CASCADE chain via `business_id` already touches 26+ tables — adding new CASCADE FKs at the process_diagrams / auth.users level needs verification that we're not accidentally chaining into that broader cascade.

Output: 1 migration (~5 FKs in 3 logical groups) + 1 test file + fk-policy.md update.
</objective>

**Cascade-chain verification: the test plan**

For each CASCADE FK, the test asserts:
1. **Immediate cascade:** child row is gone after parent delete. Use `assertOrphans(supabase, table, fkColumn, parentId, 'cascade')`.
2. **No-cross-cascade (bounded):** assert that an unrelated row in the same table (FK pointing at a DIFFERENT parent) is unaffected. This catches accidentally-too-broad CASCADEs.
3. **No-orphan-elsewhere:** for `session_attendees.user_id` specifically, assert the parent `sessions` row that the attendance pointed at still exists — the attendance going away should NOT cascade up to the session.

**Test pattern for `process_flows.from_step_id`:**

```ts
it('process_flows row CASCADEs when from_step is deleted', async () => {
  if (skipIfNoLiveDb()) return
  const supabase = getTestSupabase()
  await seedTestBusiness(supabase)

  // Setup: create a diagram, two steps, a flow between them, an UNRELATED flow.
  const { data: diagram } = await supabase.from('process_diagrams').insert({ ... }).select('id').single()
  const { data: step1 } = await supabase.from('process_steps').insert({ process_id: diagram.id, ... }).select('id').single()
  const { data: step2 } = await supabase.from('process_steps').insert({ process_id: diagram.id, ... }).select('id').single()
  const { data: flow } = await supabase.from('process_flows').insert({
    from_step_id: step1.id, to_step_id: step2.id, process_id: diagram.id,
  }).select('id').single()
  const { data: unrelatedFlow } = await supabase.from('process_flows').insert({
    from_step_id: step2.id, to_step_id: step1.id, process_id: diagram.id,
  }).select('id').single()

  // Action: delete step1.
  await supabase.from('process_steps').delete().eq('id', step1.id)

  // Assert: flow is gone (CASCADE fired).
  const { data: flowAfter } = await supabase.from('process_flows').select('id').eq('id', flow.id).maybeSingle()
  expect(flowAfter).toBeNull()

  // Assert: unrelatedFlow's to_step_id pointed at step1, so it ALSO cascades (because process_flows.to_step_id is also CASCADE).
  // OR if to_step_id is independently SET NULL, this test surfaces the disagreement.
  // Re-confirm with fk-policy.md before writing the assertion.

  // Assert: diagram still exists (cascade did NOT propagate up).
  const { data: diagramAfter } = await supabase.from('process_diagrams').select('id').eq('id', diagram.id).single()
  expect(diagramAfter.id).toBe(diagram.id)

  // Assert: step2 still exists (only step1 was deleted).
  const { data: step2After } = await supabase.from('process_steps').select('id').eq('id', step2.id).single()
  expect(step2After.id).toBe(step2.id)
})
```

The "unrelated row" check is the load-bearing one — it catches the failure mode where a CASCADE accidentally chains too far.
