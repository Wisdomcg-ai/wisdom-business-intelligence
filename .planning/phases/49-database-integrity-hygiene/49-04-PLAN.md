<objective>
First half of the DB-04 SET NULL conversions. Per `docs/db/fk-policy.md` Bucket A (signed off in plan 49-02), convert ~24 of the ~48 audit-attribution FKs from NO ACTION to ON DELETE SET NULL.

**Why this is its own plan:** PHASE.md success criterion #3 says "one-or-two FKs per migration." RESEARCH.md DB-04 reframed: "one or two FKs per migration *file*, multiple files per PR, organised by bucket" — bundling all 24 FKs in this batch into one migration with sectioned ALTER TABLE blocks lets the PR be reviewable while the test suite covers each FK individually. Splitting Bucket A across 49-04 and 49-05 lets lessons from this batch inform the second.

**Why depends on 49-02:** the policy doc must be signed off before any DB-04 migration ships. RESEARCH.md DB-04 line 408: "DB-03 sign-off by developer (Matt) required before DB-04 starts."

**Why this plan also creates the shared test helper:** RESEARCH.md DB-04 lines 421-425 recommend extracting `skipIfNoLiveDb`, `createTestUser`, `deleteTestUser`, `assertOrphans` from the existing 06C test pattern into `src/__tests__/migrations/_helpers.ts`. This plan creates the helper so it's available for plans 49-05, 49-06, 49-07 to extend rather than duplicate.

**The test pattern (RESEARCH.md DB-04 lines 427-437):** for each FK, a test that:
1. `createTestUser(supabase)` → returns `userId` via `supabase.auth.admin.createUser`
2. Inserts a dependent row in the affected table referencing `userId` (e.g., `action_items.assigned_to = userId`)
3. `deleteTestUser(supabase, userId)` → `supabase.auth.admin.deleteUser`
4. `assertOrphans(supabase, table, fkColumn, userId, 'null')` → asserts the row still exists with the FK column = NULL

**Critical:** these tests can't run in the CI vitest job (placeholder env vars at `supabase-preview.yml:113-133`). They must run locally against a Supabase preview branch with real env vars set. The PR description includes the test output (with non-skipped result) per RESEARCH.md DB-04 lines 440-446.

**Migration shape (RESEARCH.md DB-04 lines 386-401):** DROP CONSTRAINT IF EXISTS + ADD CONSTRAINT, NOT ALTER CONSTRAINT (PostgreSQL doesn't support that). Atomic within the migration's transaction.

Output: 1 migration (~24 FKs) + 1 helper file + 1 test file (~24 cases) + fk-policy.md update.

**Migration shape (per FK, RESEARCH.md DB-04 lines 386-401):**

```sql
-- Phase 49 DB-04: SET NULL on action_items audit FKs.
-- Per docs/db/fk-policy.md Bucket A row #1 + #2.
-- Tested via src/__tests__/migrations/db-04-set-null-batch-1.test.ts.

ALTER TABLE "public"."action_items"
  DROP CONSTRAINT IF EXISTS "action_items_assigned_to_fkey";
ALTER TABLE "public"."action_items"
  ADD  CONSTRAINT "action_items_assigned_to_fkey"
       FOREIGN KEY ("assigned_to") REFERENCES "auth"."users"("id")
       ON DELETE SET NULL;

ALTER TABLE "public"."action_items"
  DROP CONSTRAINT IF EXISTS "action_items_created_by_fkey";
ALTER TABLE "public"."action_items"
  ADD  CONSTRAINT "action_items_created_by_fkey"
       FOREIGN KEY ("created_by") REFERENCES "auth"."users"("id")
       ON DELETE SET NULL;
```

**DROP + ADD pattern (NOT ALTER):** PostgreSQL does not support `ALTER CONSTRAINT … SET ON DELETE`. Drop and recreate is the only path. Inside a single transaction (which migrations run in), this is atomic and safe.

**Constraint name discovery:** baseline schema names FK constraints predictably (`<table>_<column>_fkey`) but verify each name before use:
```bash
grep -nE "ADD CONSTRAINT .* FOREIGN KEY.*action_items" supabase/migrations/00000000000000_baseline_schema.sql | head
```
If any constraint deviates (e.g., custom name with explicit suffix), use the actual name — `DROP CONSTRAINT IF EXISTS` is forgiving but `ADD CONSTRAINT` re-creates with the exact name we provide.

**Per-FK test pattern:**

```ts
it('action_items.assigned_to becomes NULL when assignee is deleted', async () => {
  if (skipIfNoLiveDb()) return
  const userId = await createTestUser(supabase)
  // Insert a parent business if the table FKs to business_id (most do).
  // Insert the dependent row referencing userId via the FK column under test.
  const { data: row } = await supabase.from('action_items').insert({
    business_id: TEST_BUSINESS_ID, title: 'fk-test', assigned_to: userId,
  }).select('id').single()
  await deleteTestUser(supabase, userId)
  await assertOrphans(supabase, 'action_items', 'assigned_to', userId, 'null')
  // Cleanup: delete the dependent row to keep preview branch clean for re-runs.
})
```

Each test is idempotent (uses deterministic test IDs or cleans up after itself per the 06C convention).

**Insert dependencies (the practical wrinkle):** many of the affected tables have NOT NULL `business_id` columns FK'd to `businesses.id` or `business_profiles.id`. The test must seed a `businesses` row before inserting the dependent. The helper file should expose a `TEST_BUSINESS_ID` deterministic UUID + a `seedTestBusiness(supabase)` function used by all per-FK tests. The 06C pattern uses a uuid-v5-derived static namespace string (`00000000-0000-4000-8000-00000044206c`) — match that style.
